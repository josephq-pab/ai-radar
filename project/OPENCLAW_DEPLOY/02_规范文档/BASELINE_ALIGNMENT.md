## 2. 前期资料的正式基线

### 2.1 最小原型范围

根据《最小可运行原型范围说明》《原型页面与信息架构说明》《原型搭建任务拆解清单》，首期最小原型范围应收敛为 **4 个核心页面**：

1. 首页总览
2. 存款深案例页
3. 贷款深案例页
4. 重点事项跟踪页

### 2.2 首期核心能力

首期需要验证的不是“大而全的信息平台”，而是以下最小链路是否可跑通：

- 首页总览
- 双案例承接（存款 / 贷款）
- 重点事项闭环
- 周报 / 专题入口联动

### 2.3 首期不应扩张的范围

前期资料已明确首期暂不纳入：

- 高保真视觉稿
- 复杂权限体系
- 审批流 / 自动派单
- 自动专题生成引擎
- 全量风险模型接入

### 2.4 信息架构原则

前期资料要求采用：

- **1 个总览入口**
- **2 个业务深案例页**
- **1 个统一跟踪页**

周报 / 专题首期作为**二级承接内容**存在，而不是优先扩为独立一级页面体系。

### 2.5 首页定位

首页不是数据大屏，也不是孤立 KPI 页，而是管理型总览入口，必须同时承担：

- 全局概览
- 重点变化发现
- 业务维度导航
- 重点事项入口
- 周报与专题入口
- 待补资料 / 待处理任务提示

## 4. 差异判断：什么是“偏离”，什么是“可保留扩展”

### 4.1 属于结构偏离的部分

以下部分应视为**需要收敛**，而不是继续扩张：

#### A. `peer-monitor.html`
问题：
- 当前更像一级业务页
- 但前期资料并未把“同业监测”定义为独立一级核心页

正确归位：
- 作为首页重点摘要来源
- 作为存款页 / 贷款页中的“对标分析 / 同业动态”模块来源
- 如需保留页面，可降级为二级分析页，而不应继续增强为主入口

#### B. `weekly-report.html`
问题：
- 当前呈现为较独立的周报中心
- 但前期资料要求首期“周报 / 专题入口联动”，而非单独扩展成主体系

正确归位：
- 保留作为承接页
- 在信息架构中降级为首页 / 案例页 / 跟踪页中的入口所指向的二级页
- 暂不继续以“一级主导航页面”心态扩张

#### C. `policy-industry.html`
问题：
- 当前为预留页
- 但前期资料的首期重点仍是首页、双案例、跟踪闭环

正确归位：
- 暂停作为首期重点建设对象
- 仅保留为后续扩展占位，不纳入当前主链路

### 4.2 属于可保留的扩展能力

以下部分虽然超出最小原型，但属于**可保留、应收编复用**的能力：

- 偏好学习闭环
- 栏目模板与板块模板
- 数据解释规则
- 建议生成规则
- review / 状态汇总机制

原因：
- 这些是能力层，不必强绑定为独立页面
- 它们可以挂接到首页、案例页、跟踪页中
- 属于未来正式系统的核心资产，不应丢弃

## 7. 当前最需要新增或重构的页面

### 7.1 重点事项跟踪页是当前缺口

前期资料明确要求 4 个核心页之一是“重点事项跟踪页”，而当前工作区尚未形成一个正式、清晰的 `tracking.html` 主页。

这意味着：
- 页面主链路还不完整
- review / status 虽有数据能力，但没有真正归位到“闭环主页面”

因此，**下一轮页面施工的第一优先级，不是继续扩周报中心，而是补齐 tracking.html。**

### 7.2 首页与深案例页需要重新按基线校正模块

- 首页：强化“总览 + 跳转 + 事项入口 + 周报入口”角色
- 存款页：更明确变成深案例页，而不只是数据页
- 贷款页：更明确变成深案例页，而不只是指标页

## 8. 建议的下一轮施工顺序

### 第一阶段：结构归位
1. 输出本基线对齐方案
2. 新增 / 重构 `tracking.html`
3. 调整首页、存款页、贷款页、周报页之间的主跳转关系
4. 明确 `weekly-report.html` / `peer-monitor.html` 的二级定位

### 第二阶段：内容归位
1. 首页补齐重点事项摘要与周报 / 专题入口
2. 存款页补齐“关键观察 / 建议动作 / 跟踪事项摘要”
3. 贷款页补齐“监管雷达 / 行业风险 / 建议动作 / 跟踪事项摘要”
4. 跟踪页挂接 reviewQueue / reviewStatus / 下一步动作

### 第三阶段：规则归位
1. 将观察提示优先挂到首页与案例页摘要区
2. 将策略建议优先挂到案例页“建议动作”区
3. 将直接行动建议与待人工确认事项挂到跟踪页
4. 将周报页降为规则透明化与草稿承接页

---

## 9. 当前不建议继续优先推进的事项

以下事项不建议作为当前第一优先级继续投入：

- 继续扩周报中心为更完整独立系统
- 继续增强 `peer-monitor.html` 为一级强入口页
- 提前把政策 / 行业页做深
- 在页面数量上继续膨胀

原因：
- 容易偏离首期最小原型目标
- 会削弱首页—双案例—跟踪闭环主链路
- 会让验收标准变模糊

---

## 10. 结论

当前项目**不是做错了**，而是已经从“最小原型验证”自然长到了“功能雏形扩展”阶段。

现在最正确的动作不是回退，也不是继续扩张，而是：

- 用前期资料把结构重新校准
- 把已实现能力收编到正确页面架构中
- 先补齐“重点事项跟踪页”这个主链路缺口
- 再继续推进规则如何挂接到首页 / 案例页 / 跟踪页

一句话总结：

> 当前最优先任务，不是再造新页面，而是把已做出的能力收敛进“首页总览—存款案例—贷款案例—重点事项跟踪”这条正式最小链路。
