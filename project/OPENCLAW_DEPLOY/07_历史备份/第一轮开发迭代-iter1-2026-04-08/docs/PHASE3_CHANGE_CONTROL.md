# Phase 3 变更控制清单（首版）

> 文档状态：初稿 v0.1
> 起草时间：2026-04-07
> 用途：防止功能膨胀和方向漂移

---

## 一、变更控制原则

Phase 3 性质为**收敛稳定**，非扩展阶段。任何新想法按以下三类处理，不直接进入主线：

| 类别 | 处理方式 |
|------|---------|
| 当前必须做 | 纳入 P0/P1 任务包，进入主线 |
| 后续可做 | 记录在本清单"后续可考虑"节，不进入当前主线 |
| 当前明确不做 | 记录在本清单"当前明确不做"节，说明理由 |

---

## 二、当前明确不做事项

以下事项在 Phase 3 期间**明确不做**，不受任何"顺手"理由驱动：

### 2.1 Gate A 数据修复
- **不做原因**：依赖外部数据供给，系统层无解
- **处理**：文档标注，暂挂
- **后续条件**：外部 Excel 到位后执行

### 2.2 重型后台建设
- **不做原因**：Phase 3 为 MVP 闭环，不需要完整后台
- **后续条件**：试点运行 3 个周期后，如数据量/人工作业量确实不可接受，再评估

### 2.3 复杂权限体系
- **不做原因**：当前无多角色需求，权限体系是过度建设
- **后续条件**：角色视图（P2-2）明确后，如存在真实多角色需求，再评估

### 2.4 未列入 P0/P1 的功能新增
- **不做原因**：Phase 3 阶段边界已定义，功能扩展不属于本阶段目标
- **处理**：任何新想法先记录→评估是否属于当前 sprint→不属于则进入 backlog

### 2.5 微信公号抓取方案（主动推进）
- **不做原因**：苏商银行来源已可覆盖部分需求；wechatsogou 存在安装障碍；微信平台壁垒在无外部依赖下不可解
- **处理**：记录在"后续可考虑"，条件成熟时可重新评估

### 2.6 界面精致化
- **不做原因**：Phase 3 优先级为结构治理，界面精致化是锦上添花
- **后续条件**：MVP 闭环稳定运行后，再评估

### 2.7 跨项目借用文件/代码
- **不做原因**：边界必须清晰，防止数字员工项目污染雷达站
- **处理**：如发现类似需求，明确说明跨项目边界，请求用户确认

### 2.8 过早的完整性建设
- **不做原因**：Phase 3 目标是可演示，不是有完整产品
- **示例**：不因为"应该有数据 dashboard"就做一个完整 dashboard；先确认"有没有数据可展示"

## 四、变更申请流程

如用户提出新想法，优先判断：
1. **是否属于 Phase 3 五项推进点**：是 → 评估是否纳入 P0/P1；否 → 归入 backlog
2. **是否影响当前 MVP 闭环**：是 → 评估紧急程度；否 → 进入 backlog
3. **是否有外部依赖**：是 → 标注条件，不作为当前任务推进

## 六、审查机制

- **每 sprint review**：每个 sprint 结束时，检查变更控制清单是否需要更新
- **每月 review**：如果清单中"后续可考虑"事项超过 5 项，重新评估是否需要提升优先级
- **异常触发**：如果某 sprint 完成的功能超出任务包范围，必须在 review 纪要中说明并修正
