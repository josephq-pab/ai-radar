# Phase 3 任务包

> 文档状态：初稿 v0.1
> 对应阶段：Phase 3 — 试点运营化 / MVP闭环
> 起草时间：2026-04-07

---

## 概述

本任务包将 Phase 3 目标拆解为 P0 / P1 / P2 三个优先级，共 8 个任务。

**优先级定义**：
- P0：MVP 闭环必需，当前 sprint 必须完成
- P1：MVP 可运行，但可稍后；有助于提升可用性
- P2：本阶段不做，记录在案，后续可考虑

**每项任务格式**：
- 任务目的
- 对应推进点（输出分层 / 确认项分级 / 周期管理 / 角色视图 / 数据源轻管理）
- 输入
- 输出
- 依赖
- 验收方式
- 是否阻断试点

---

## P0 任务（共 3 项）

---

### P0-1：前端页面试点运行验证

**任务目的**：验证前端页面 + 最近一次抓取结果可正常展示，数据 freshness 可确认。

**对应推进点**：输出分层、周期管理

**输入**：
- 前端：已修复的 index.html（路径 03_前端页面/）
- 后端：最近一次抓取的 analyst_opinions_raw.json

**输出**：
- 前端页面可访问（http://47.112.211.98:8787）
- 页面显示最近抓取的分析师范畴、数据条数、最新抓取时间
- 如有问题，记录问题并定位

**依赖**：
- M1（运行入口稳定）：✅ 已完成
- M2（分析师抓取可运行）：✅ 已完成

**验收方式**：
- 人工访问 http://47.112.211.98:8787/index.html
- 确认页面可加载、无空白、无报错
- 确认可看到最新抓取数据摘要

**是否阻断试点**：是

---

### P0-2：分析师观点输出分层固化

**任务目的**：明确分析师抓取结果的输出分层（raw → reviewed → report-ready），确认每层的格式和负责人。

**对应推进点**：输出分层

**输入**：
- analyst_opinions_raw.json（当前抓取结果）
- build_analyst_opinions.py（中间层构建脚本）
- review 流程文档（如有）

**输出**：
- 明确三层输出的定义和字段
- 确认 build_analyst_opinions.py 当前状态（是否能跑通到 reviewed 层）
- 确认 reviewed 层到 report-ready 层的转化逻辑是否已实现
- 输出文档：analyst_opinions.json（reviewed 层）和 report_ready.json（report-ready 层）

**依赖**：
- P0-1（M4 试点运行验证）

**验收方式**：
- `build_analyst_opinions.py --dry-run` 可执行
- analyst_opinions.json 生成非空记录
- 文档中明确三层各自包含哪些字段

**是否阻断试点**：是

---

### P0-3：周期运行脚本固化

**任务目的**：将分析师抓取固化为一键可执行脚本，明确运行周期和失败处理。

**对应推进点**：周期管理

**输入**：
- fetch_analyst_articles.py（当前版本）
- analyst_sources.json（已更新的来源配置）

**输出**：
- 一条可执行的周期运行命令（写入 RUNBOOK.md）
- 明确运行频率（如：每周一执行）
- 明确失败时的处理（重试次数、报警方式或跳过机制）
- 明确数据过期处理（多久之前的数据视为过期）

**依赖**：
- P0-2（输出分层确认后，才能定义什么算"成功运行"）

**验收方式**：
- 运行一次完整抓取，确认 analyst_opinions_raw.json 有新增记录
- 记录本次运行耗时和成功率
- RUNBOOK.md 包含完整运行命令

**是否阻断试点**：是

## P1 任务（共 3 项）

---

### P1-1：试点运行状态追踪表

**任务目的**：建立轻量级追踪表，记录每次试点运行的结果（抓取条数、成功/失败/跳过、耗时）。

**对应推进点**：确认项分级、周期管理

**输入**：历次 fetch_analyst_articles.py 运行结果

**输出**：
- CSV 或 JSON 格式的运行日志（processed/fetch-run-log.json 或 .csv）
- 字段：run_date, source, total, success, skipped_old, failed, duration_sec
- 一条记录代表一次完整抓取运行

**依赖**：P0-3

**验收方式**：
- 连续运行 2 次抓取，日志均有记录
- 日志可查询、可排序

**是否阻断试点**：否（有助于追踪，但手动记录也可接受）

---

### P1-2：分析师来源有效性 review

**任务目的**：对当前 13 个分析师来源的有效性做一次 review（维度相关性、URL 可访问性、内容质量）。

**对应推进点**：输出分层、数据源轻管理

**输入**：analyst_sources.json（当前 13 个来源）

**输出**：
- review 结果（哪些来源值得保留，哪些降级或移除）
- 更新的 analyst_sources.json（如有变更）
- 文档记录：analyst_review_notes.md

**依赖**：抓取结果数据（analyst_opinions_raw.json）

**验收方式**：
- review 结果有记录
- 有明确结论（保留/降级/移除）

**是否阻断试点**：否

---

### P1-3：确认项分级规则文档化

**任务目的**：将"什么算 confirmable / referenceable / reportable"形成明确规则文档，供 review 使用。

**对应推进点**：确认项分级

**输入**：历史 review 决策（analyzable 表中的 reviewDecision 字段）

**输出**：
- docs/CONFIRMABILITY_RULES.md（或在现有 EVAL_RULES.md 中补充）
- 明确三类确认项的定义和判断标准

**依赖**：analyzable 表中有足够的历史 review 记录

**验收方式**：
- 文档存在且具体
- 给出一个示例判断（某条记录属于哪类，说明理由）

**是否阻断试点**：否
