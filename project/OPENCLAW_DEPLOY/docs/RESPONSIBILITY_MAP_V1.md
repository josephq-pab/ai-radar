## 一、使用说明

本表的角色级建议是基于**典型商业银行组织架构**推断的，适用于：
- 分行公司业务部门
- 分行客户管理部门
- 战略研究 / 企金部门
- 资产负债 / 计财部门
- 总行产品部门

**使用原则**：
1. 角色级建议仅供参考，不强制匹配任何具体银行的具体岗位
2. 同一动作在不同银行可能由不同部门/角色承担
3. 本表目的是让建议"看起来像真实分工"，不是"要求某行必须这样做"
4. 使用时可根据实际反馈调整

## 二、动作类型与责任归口总表

### 2.1 立即执行动作

| 动作类型 | 建议部门 | 建议角色 | 说明 |
|---------|---------|---------|------|
| 大额客户异动排查（挖转风险）| 分行公司部 / 分行客户管理部 | 客户经理（主办）+ 部门负责人（督办）| 主办是实际执行人，督办是兜底人 |
| 重点客户联系 / 到期提醒 | 分行客户管理部 / 零售部 | 客户经理（主办）+ 客服/运营（协办）| 若存款归对公，则为分行公司部客户经理 |
| 客户流失风险排查 | 分行公司部 | 客户经理（主办）+ 产品经理（支持）| 客户经理是发现端倪的人，产品经理是分析的人 |
| 外部利率变化监测（做一次）| 分行公司部 / 总行资产负债部 | 产品经理（主办）+ 客户经理（信息收集）| 一次性的监测动作，不是建立机制 |
| 内部数据核实 / 调取 | 相关部门 | 对应岗位（视数据来源定）| 不单独作为主动作，一般是其他动作的前置步骤 |

### 2.2 研究准备动作

| 动作类型 | 建议部门 | 建议角色 | 说明 |
|---------|---------|---------|------|
| 竞争对手策略研究（存款/贷款）| 战略研究部 / 企金部 | 行业研究员（主办）+ 企金部产品经理（支持）| 研究结论接收方通常是部门负责人或分行长 |
| 宏观政策影响评估 | 战略研究部 / 资产负债部 | 宏观经济研究员（主办）| 若宏观组设在战略部，则为战略部；若设在资产负债部，则为该部 |
| 客户异动深度研究 | 分行公司部 / 客户管理部 | 客户经理（主办）+ 战略研究员（支持）| 深度研究需要一线信息+分析能力配合 |
| 产品竞争力分析（存款/贷款）| 企金部 / 产品部门 | 产品经理（主办）+ 客户经理（信息收集）| 需要产品定价数据+客户反馈数据 |
| 利率走势判断 | 资产负债部 / 计财部 | 利率管理岗（主办）| 需要资产负债管理视角 |

### 2.3 观察项（执行人设定）

| 观察类型 | 建议部门 | 建议角色 | 说明 |
|---------|---------|---------|------|
| 客户异动信号观察 | 分行公司部 / 客户管理部 | 客户经理（日常工作中自然承担）| 不需要额外安排时间，是工作本能 |
| 外部利率变化跟踪 | 总行资产负债部 / 企金部 | 利率管理岗 / 产品经理 | 日常工作的一部分 |
| 竞争对手动态跟踪 | 战略研究部 / 企金部 | 行业研究员（日常跟踪）| 战略部的正常工作内容 |
| 市场数据跟踪 | 战略研究部 / 资产负债部 | 宏观经济研究员 / 数据分析岗 | 日常工作 |

## 三、角色级建议的最小粒度说明

### 3.1 什么情况下可以写到角色级

以下情况，角色级建议可以更具体：

1. **动作是某角色的本职工作**：客户经理排查大额客户异动，就是客户经理的活
2. **动作需要特定专业能力**：宏观政策影响评估，就是宏观经济研究员的活
3. **动作主体在各行相对稳定**：利率走势判断，基本上是资产负债部的活

### 3.2 什么情况下不能写死，只能写"相关部门/角色联合"

以下情况，组织架构差异太大，角色级写太细反而武断：

1. **客户挖转风险排查**：有的银行是对公客户经理的活，有的银行是客户管理部的活，有的银行是分行长的活
2. **产品竞争力分析**：有的银行是产品经理的活，有的银行是企金部的活，有的银行是战略部的活
3. **存款策略研究**：有的银行是战略部的活，有的银行是资产负债部的活

**处理方式**：写"主办+支持"双角色，不指定具体谁主办

```
建议写法：分行公司部（主办）+ 战略研究部（支持）
不推荐的写法：战略研究部主办，分行执行
```

### 3.3 不能写的角色级（武断风险高）

| 动作 | 武断原因 | 正确处理 |
|------|---------|---------|
| "谁是大客户"的具体定义 | 各行标准不同 | 不写具体金额，写"5000万以上客户"并注明"以各行认定为准" |
| "分行长亲自盯" | 分行长不分管具体业务 | 写"分行公司部负责人（或分行长）" |
| "总行资产负债部主办" | 很多银行这个职能在计财部 | 写"资产负债/计财部门" |
| "客户经理A负责" | 具体人名是派工单的事 | 只写"客户经理" |

## 四、研究准备动作的责任链路

研究准备动作的特点是：研究责任人 ≠ 产出接收方。这两个角色之间的信息传递必须明确。

**标准责任链路**：
```
研究责任人（研究员/产品经理）
    ↓ 产出（报告/表格/备注）
接收方（部门负责人/分行长）
    ↓ 判断（是否触发转动作条件）
    ├─ 触发 → 下月执行动作（责任人：主办部门）
    └─ 不触发 → 归档，结论留存
```

**特别说明**：
- 研究责任人负责"做研究"，接收方负责"看完决定要不要动"
- 两者之间不需要审批流程，接收方看完直接决定
- 若接收方超过预计时点未给出判断，研究责任人应主动上报（触发条件里写明）

## 五、责任归口表的典型表述示例

### 5.1 立即执行动作

```
建议责任人：分行公司部 / 客户管理部
角色建议：客户经理（主办）+ 部门负责人（督办）
说明：主办是实际执行和发现端倪的人；督办是部门负责人，用于结果汇报兜底
```

### 5.2 研究准备动作

```
建议责任人：战略研究部
角色建议：行业研究员（主办）
产出接收方：分行公司部负责人（或分行长）
说明：研究员产出判断报告，接收方负责看完后决定是否触发下月动作
```

### 5.3 联合行动（主办+支持）

```
建议责任人：分行公司部（主办）+ 战略研究部（支持）
角色建议：客户经理（主办）+ 行业研究员（支持）
说明：主办负责一线信息和执行，支持负责分析能力
产出接收方：分行公司部负责人
```

### 5.4 不确定时的降调写法

```
建议责任人：[分行公司部 / 客户管理部]（注：各部门均可承接，视各行实际情况定）
角色建议：客户经理（注：若大客户归对公，则为对公客户经理；若归零售，则为零售客户经理）
说明：建议在试用前与对应部门确认承接角色
```

## 七、使用检查表

在填写完动作责任归口后，用以下问题自检：

1. **责任是否到部门+角色？** → 不能只写"相关部门"
2. **是否区分了主办和支持角色？** → 联合行动必须写主办+支持，不能只写主办
3. **研究准备的接收方是否写了？** → 不能留空
4. **接收方是否是"能决定要不要动"的人？** → 不能是下属角色（下属没有决策权）
5. **角色级建议是否在武断风险可接受范围内？** → 遇到组织架构差异大的情况，是否已加"注"说明
6. **是否避免了写具体人名？** → 不能写"张三负责"，只能写"客户经理负责"
