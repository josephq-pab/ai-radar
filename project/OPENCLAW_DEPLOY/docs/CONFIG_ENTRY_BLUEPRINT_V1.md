## 一、推荐方案：静态配置状态页型

**方案定性**：纯前端只读配置展示页，不做后端/数据库/编辑能力。

---

## 二、页面/入口最小形态

### 入口形式

**推荐**：新建 `config-status.html`，放在 `03_前端页面/` 目录下，与 index.html 同级。

**备选**：集成到 index.html 作为 tab切换（当前不选，等独立页面验证稳定后再考虑集成）。

**理由**：独立页面更容易维护，不影响现有 index.html 的稳定性。

### 访问方式

- 直接在浏览器打开 `config-status.html`（file:// 协议可用）
- 从 index.html 侧边栏添加"配置状态"导航入口
- 无需服务器，无需登录

### 技术实现

- 纯 HTML + 内联 JavaScript
- JavaScript 直接读取 `../data/analyst_sources.json` 等 JSON 文件（浏览器安全限制需要通过 Web server 访问，或者用内嵌 JSON 方式）
- **当前实现方式**：内嵌 JSON 快照（手动刷新脚本输出），不依赖跨域读取

**简化做法**：配置值直接写在 HTML 里作为快照，每次配置变更后手动刷新快照。

## 三、展示字段设计

### 每项配置的展示格式

```
配置名称    [配置名称]
当前值      [当前值]
来源文件    [来源文件:行号]
生效说明    [值的含义]
最近变更    [变更时间] [变更内容]
─────────────────────────────────
```

### 配置分组展示

**分组1：来源配置**
- 激活来源列表
- 停用来源及原因

**分组2：Gate 状态**
- Gate A（数据新鲜度）
- Gate B（内容可用性）
- Gate C（跟踪就绪度）

**分组3：运行参数**
- top-k 默认值
- B gate 阈值
- 当前运行模式

**分组4：内容基线**
- 当前内容基线版本
- 当前试用对象
- 当前试用周期

## 五、当前不做的能力

| 不做 | 原因 |
|------|------|
| 配置编辑器（在线改配置）| 风险高，容易破坏 JSON 结构 |
| 后端/API | 过渡建设，MVP 不需要 |
| 数据库存储 | CSV/JSON 足够 |
| 多环境配置分离 | dev/prod 分离是正式部署需求 |
| 自动化告警 | 当前人工月度节奏足够 |
| 配置历史版本对比 | 第一期只要求有变更记录，不要求完整历史 |
| 实时监控 | 第一期只要求手动刷新 |

## 七、实施注意事项

1. **不修改任何现有 JSON 文件**：配置状态页是只读展示
2. **不引入新的依赖**：纯 HTML + 内联 JS，不安装 npm 包
3. **每次配置变更后手动刷新快照**：在 config-change-log.md 里记录
4. **不影响现有页面**：config-status.html 是独立的，不改 index.html
