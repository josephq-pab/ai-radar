## 背景

P3-33 已完成值班结果四分类最小输出模板，运营者现在知道"无动作 / 仅记录 / 进入评估 / 进入准备"各怎么写。

但仍存在一个问题：

> **当结果进入评估/准备时，何时应升级为人工拍板请求？怎么写升级说明？**

当前摩擦：
- 轻微信号被写成升级项（评估请判断≠所有评估项都要升级）
- 进入评估就默认一定要请人拍板
- 进入准备和正式运行被混淆（准备≠触发 RUN-next）
- 升级时材料过长、重点不清
- 升级后还在继续补很多页，无法收口

本功能包定义升级条件、升级层级、最小升级说明模板。

---

## 升级层级定义

### A. 不升级（默认）

**适用条件**：无动作 / 仅记录 / 轻微信号 / 口径澄清 / 仅进入评估但边界清晰

**默认动作**：按值班输出模板（P3-33）留痕即可，不发起拍板请求

**不应做的事**：
- 把轻微信号写成升级说明
- 把"仅记录"写成"请判断"
- 把边界清晰的评估结论写成请拍板

**典型不升级情形**：
| 情形 | 类型 | 动作 |
|------|------|------|
| 快报告知"等待态正常" | 无动作 | 不升级，一句话留痕 |
| 口径澄清（宏观分析=政策影响） | 仅记录 | 不升级，2~3行留痕 |
| 1次轻微信号，无持续性 | 轻微信号 | 不升级，记录即可 |
| usable 池小幅波动（<10%） | 轻微信号 | 不升级，观察即可 |
| 评估边界清晰，已给出结论 | 进入评估 | 不升级，直接留评估结论 |

---

### B. 可升级为"评估请判断"

**适用条件**：边界不清 / 触发判断不清 / 异常待判 / 结构性问题待决

**默认动作**：给出最小评估说明，请人判断是否继续

**何时考虑升级为"评估请判断"**：
- usable 池明显异常（下降 >30%），不确定是否触发
- 触发条件边界模糊（是否命中不明确）
- 异常信号连续出现（2次以上），需要判断是否进入准备
- 有提议提前跑 RUN-next，需要拍板是否允许
- 结构性输入出现，需要判断是否进入第二链路
- 其他边界不清、需要人工判断的事项

**不应做的事**：
- 进入评估就默认要升级
- 把判断清晰的评估写成请拍板
- 升级后继续写很多补充材料

---

### C. 可升级为"准备请确认"

**适用条件**：明显新信号已出现 / 运行前条件接近满足 / 已进入准备但尚未运行

**默认动作**：给出准备说明，请确认是否允许进入下一步准备

**何时考虑升级为"准备请确认"**：
- 曾刚连续断档 2 个月以上，需要决定是否解冻
- usable 池持续低于阈值（<5 条），需要决定是否触发
- 触发条件已明确命中，需要确认是否可以开始准备
- 其他需要人工确认才能进入下一步的情形

**不应做的事**：
- 把"准备"写成"RUN-next 已触发"
- 把准备当成正式运行
- 升级后继续写运行进度报告

## 升级说明最小字段

每类升级说明的必须字段：

| 必须字段 | 评估请判断 | 准备请确认 |
|---------|----------|----------|
| 当前结果类型 | ✅ | ✅ |
| 为什么需要升级 | ✅ | ✅ |
| 当前还未做什么 | ✅ | ✅ |
| 希望对方拍板什么 | ✅ | ✅ |
| 最小必要依据（在哪） | ✅ | ✅ |

## 升级说明最小模板

### B. 评估请判断（最小模板，3~5行）

```
[日期] 评估请判断。

【当前结果】：进入评估
【升级原因】：[什么信号/边界不清/判断困难]
【当前未做什么】：[尚未给出触发结论，等待判断]
【希望拍板】：[请判断是否触发 / 请判断是否进入准备]
【最小依据】：[在哪查：ops-evidence.html / ops-decision.html / 真源字段]
```

**示例**：

```
[日期] 评估请判断。

【当前结果】：进入评估
【升级原因】：usable 池较上周下降 35%，不确定是否达到触发阈值
【当前未做什么】：尚未给出触发结论，等待判断
【希望拍板】：请判断是否触发 RUN-next，或是否需要进一步准备
【最小依据】：ops-evidence.html → usable 池趋势 / STAGE2_SINGLE_CHAIN_TRIGGER_RULES_V1.md → 触发条件定义
```

---

### C. 准备请确认（最小模板，3~5行）

```
[日期] 准备请确认。

【当前结果】：进入准备
【升级原因】：[什么信号表明需要人工确认]
【当前未做什么】：[尚未正式触发 RUN-next，等待确认]
【希望拍板】：[请确认是否可以开始准备 / 请确认触发条件是否成立]
【最小依据】：[在哪查：ops-evidence.html / 触发规则文档 / 曾刚状态]
```

**示例**：

```
[日期] 准备请确认。

【当前结果】：进入准备
【升级原因】：曾刚连续断档 3 个月，usable 池持续低于 3 条，已接近冻结条件
【当前未做什么】：尚未正式触发 RUN-next冻结程序，等待确认
【希望拍板】：请确认是否正式进入冻结程序，或是否有其他处理方式
【最小依据】：ops-evidence.html → 曾刚断档记录 / ops-registry.html → 冻结条件定义
```

## 一句话速查

| 升级层级 | 一句话格式 |
|---------|----------|
| 不升级 | `[日期] 本次不升级。按值班留痕即可。` |
| 评估请判断 | `[日期] 评估请判断：[原因]。请判断是否继续。` |
| 准备请确认 | `[日期] 准备请确认：[原因]。请确认是否可以开始下一步。` |

## 升级后停点规则

| 升级类型 | 升级后应停在 | 不应做的事 |
|---------|------------|----------|
| 评估请判断 | 等待对方判断结论 | 继续补材料/继续深查其他页面 |
| 准备请确认 | 等待对方确认或拒绝 | 写运行进度报告/触发脚本/自行决定 |

**原则**：升级说明是请求判断，不是汇报进展。升级后由接收方给出结论，运营者不应继续自主扩写。

---

## 什么不值得升级

| 情形 | 为什么不升级 | 正确动作 |
|------|------------|---------|
| 轻微信号（1次/小幅/无持续性）| 不影响判断，不需要拍板 | 仅记录 |
| 快报"等待态正常" | 状态清晰，不需要判断 | 无动作留痕 |
| 口径澄清 | 不是触发判断问题 | 仅记录 |
| 评估边界清晰，已给出结论 | 判断已做完，不需要再拍板 | 直接留评估结论 |
| 预防性判断"会不会出事" | 没有信号不应评估 | 不评估，不升级 |
| 想"再确认一下才放心" | 没有实质判断需求 | 停止，不要继续翻页 |

## 升级说明常见混淆

| 混淆点 | 正解 |
|--------|------|
| 进入评估就一定要升级 | ❌ 评估边界清晰时不需要升级 |
| 升级说明越长越稳妥 | ❌ 最小必要字段即可，不需要长篇大论 |
| 升级后继续补材料 | ❌ 升级后停在等待，不继续扩写 |
| 进入准备等于 RUN-next 已触发 | ❌ 准备是"准备开始"，触发是"正式开始"，不同阶段 |
| 轻微信号也写成请判断 | ❌ 轻微信号属于"仅记录"，不需要请拍板 |
| 升级时贴很多依据更稳妥 | ❌ 最小必要依据即可，不需要贴全部证据 |

---

## 当前不应误解的事项

1. **评估≠升级**：进入评估不等于一定要升级。只有边界不清/判断困难时才升级。
2. **准备≠触发**：进入准备不等于 RUN-next 已触发。准备是"准备开始"，触发是"正式开始"。
3. **升级说明是请求判断，不是汇报**：升级后应停在等待方结论，不继续写补充材料。
4. **大多数值班不需要升级**：只有满足 B 或 C 条件时才升级，A 类默认不升级。
5. **升级说明有最小字段，但不是全部字段**：最小5字段是基本要求，不是完整模板。
