# OPS_MAINTENANCE_SOP_V1.md — 防漂移维护标准操作手册

> 文档版本：v1.0（P3-41）
> 创建时间：2026-04-21
> 对应阶段：Phase 3 — 试点运营化 / P3-41 全站治理底座、可维护性矩阵与防漂移维护手册功能包第一期
> 目的：建立"后续任何改动都应遵守"的标准维护步骤，让改动可追踪、不漏同步、不漂移

## 一、维护原则（每次改动前必须先读）

### 六大维护原则

**原则1：不新增入口页优先**
任何改动都优先在现有页面/文档内解决，不通过新增页面解决。

**原则2：先改真源，再改展示层**
改动时先确认是否涉及真源层（STAGE2/V7基线/analyst_sources.json），真源变了展示层必须跟。

**原则3：先统一口径，再改例子**
若改动涉及标准句/定义，先在 canonical 句库（OPS_COPY_BASELINE）统一定义，再改各页面示例。

**原则4：改页面前先查影响矩阵**
每次改页面之前，先查 OPS_CHANGE_IMPACT_MATRIX_V1.md，确认必须同步哪些其他对象。

**原则5：改完后必须同步跟踪文档**
每次改动无论大小，必须在 CHANGE_CONTROL.md 追加记录，在 REVIEW_LOG.md 留痕。

**原则6：训练层不得反向改动真源/展示层**
OPS_TRAINING_PACK / OPS_MISUSE_GUARDRAILS / OPS_DRILL_PACK 是解释性文档，发现不一致时，应以展示层为准修改训练包，而不是修改展示层。

## 二、标准维护步骤（7步）

### 第1步：判断改动层级

```
问自己：这个改动涉及哪个层级？
```

- 若涉及"对公 AI 雷达站 · V7 基线冻结"这句主线口号 → 第一层（真源层）
- 若涉及生命周期定义、阶段流转规则、升级规则 → 第一层（真源层）
- 若涉及页面内容、元信息、模块 → 第二层（展示层）
- 若涉及训练路径、误用场景、演练题 → 第三层（训练层）

### 第2步：查影响矩阵

```
问自己：改这个，必须同步哪些对象？
```

查 OPS_CHANGE_IMPACT_MATRIX_V1.md，列出"必须同步"和"建议检查"的对象清单。

### 第3步：先改真源，再改展示层，最后检查训练层

```
顺序：真源层 → 展示层 → 训练层
```

- 真源层改了，展示层必须全部检查
- 展示层改了，训练层必须检查是否需要同步
- 训练层发现不一致，只能改训练层，不能反向改展示层/真源层

### 第4步：改动前先确认canonical句

```
问自己：这个改动涉及全站标准句吗？
```

涉及标准句时，先改 OPS_COPY_BASELINE_V1.md，再改各页面。

全站标准句包括：
- 状态主线句
- 七阶段写法（W/L/A/P/E/R/S）
- 高频误判澄清句（准备≠已运行、候选≠解冻、补材料≠否决等）
- 页面角色定义句

### 第5步：执行改动

按第2步清单逐个改动，每改一个记录一个。

### 第6步：改动后验收

- [ ] 全站10个HTML页面 footer 一致性检查
- [ ] 涉及改动的模块在所有页面的一致性检查
- [ ] 交叉引用是否正常（跳转不失效）
- [ ] 训练包/误用防护/演练题是否需要同步
- [ ] STAGE2 是否需要更新

### 第7步：同步跟踪文档

- [ ] CHANGE_CONTROL.md 追加本次改动记录
- [ ] REVIEW_LOG.md 更新版本号和复盘记录
- [ ] FEEDBACK_TRACKING_STATUS.md 更新（如涉及状态变化）
- [ ] STAGE2_SINGLE_CHAIN_STATUS.md 追加（如涉及重大改动）

## 三、不同类型改动的最小同步清单

### 类型A：只改展示层内容（不改真源、不改标准句）

最小同步：CHANGE_CONTROL + REVIEW_LOG

### 类型B：改了标准句/canonical句

最小同步：
- OPS_COPY_BASELINE.md
- 全站10个HTML页面 footer
- CHANGE_CONTROL + REVIEW_LOG

### 类型C：改了生命周期/升级/回复回写规则

最小同步：
- STAGE2_SINGLE_CHAIN_STATUS.md
- 相关HTML页面（按影响矩阵）
- 相关文档包（按影响矩阵）
- OPS_LIFECYCLE_MAP 或 OPS_ESCALATION_PACK 等
- CHANGE_CONTROL + REVIEW_LOG

### 类型D：改了训练/误用/演练内容

最小同步：
- OPS_TRAINING_PACK / OPS_MISUSE_GUARDRAILS / OPS_DRILL_PACK
- 十页训练提示模块
- CHANGE_CONTROL + REVIEW_LOG

### 类型E：改了第二链路状态

最小同步：
- STAGE2_SINGLE_CHAIN_STATUS.md
- radar-home.html / single-chain-ops.html / config-status.html
- OPS_TRAINING_PACK（来源候选≠解冻）
- OPS_MISUSE_GUARDRAILS（来源候选≠解冻）
- CHANGE_CONTROL + REVIEW_LOG

## 四、什么情况下停止、不继续扩改

以下情况应停止改动，回到最小可行方案：

**情况1：改动涉及真源层（STAGE2/V7基线）**
→ 先评估影响范围，如果影响太大，优先只改展示层

**情况2：改动需要新增页面**
→ 不做，强制在现有页面内解决

**情况3：改动需要改脚本/analyst_sources.json**
→ 不做，这是真源层改动且需要额外验证

**情况4：改动需要进入RUN-02**
→ 不做，冻结状态下只做展示层/治理层改动

**情况5：改动会破坏现有训练包/误用防护的一致性**
→ 不做，必须先同步训练包再进行改动

**情况6：改动影响超过5个页面+3个文档**
→ 先做影响评估，优先做最小子集改动

## 五、全站硬约束清单（绝对不能违反）

以下内容在维护时属于"绝对不能改歪"的硬约束：

| 硬约束 | 为什么不能改 |
|--------|------------|
| 状态主线句 | 使用者对系统的第一认知，乱了等于系统失去身份 |
| W/L/A/P/E/R/S七阶段写法 | 全站协作基础，乱了等于所有人无法对齐 |
| 准备态(P)≠RUN-next已触发 | 违反会跳过关键人工确认步骤 |
| 来源候选≠解冻 | 违反会破坏口径一致性 |
| 补材料≠否决 | 违反会错误放弃有效输入 |
| 暂不推进≠否决 | 违反会错误放弃有效输入 |
| 证据页不是默认入口 | 违反会倒置判断路径 |
| 决策页不是默认入口 | 违反会浪费决策资源 |
| 第二链路冻结状态 | 违反会越界引入未经验证来源 |
