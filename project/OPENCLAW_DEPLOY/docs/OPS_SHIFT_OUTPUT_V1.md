## 背景

P3-32 已完成日常值班最小巡检路径，定义了"无动作 / 仅记录 / 进入评估 / 进入准备"四类值班结果。

但仍存在一个问题：

> **巡检完以后，本次结果到底怎么写？**

当前摩擦：
- 巡检完不知道写什么
- "无动作"写得过长（实际上只需要一句话）
- "仅记录"与"进入评估"混在一起
- 有人把轻微信号写成升级事项
- 每个人的值班说明格式都不一样

本功能包定义值班结果四分类的最小输出模板。

---

## 值班结果四分类最小输出模板

### A. 本次无动作

**适用条件**：无新数据 + 无新触发 + 无新异常 + 无新输入 + 快报结论"等待态正常"

**最小输出格式（一句话）**：

```
[日期] 本次值班无动作。RUN-next 未触发，系统处于正常等待态。
```

**扩展版（如需交代背景）**：

```
[日期] 本次值班无动作。
- 当前状态：RUN-next 未触发 / 第二链路冻结 / V7 冻结 / usable 32条
- 快报结论：等待态正常
- 本次无需进入任何评估或准备
```

**不应写成**：
- 长篇大论的状态分析
- 列出所有冻结项的详细说明
- "看起来一切正常但我还是想记录一下"

---

### B. 本次仅记录

**适用条件**：有轻微口径澄清 / 术语疑问 / 回访确认 / 轻响应输入，但不改变状态判断

**最小输出格式（2~3行）**：

```
[日期] 本次仅记录。
- 记录内容：[口径澄清内容 / 术语修正内容 / 回访确认内容]
- 状态影响：无，不改变当前等待态
- 下一步：无
```

**示例**：

```
[日期] 本次仅记录。
- 记录内容：曾刚口径澄清——"宏观分析"在本系统指政策影响类，非经济数据类
- 状态影响：无，不改变 usable 池和等待态结论
- 下一步：无
```

**不应写成**：
- 写成"进入评估"（没有触发判断问题）
- 超过 5 行的详细说明
- 附带状态变化结论（无变化则不写变化）

---

### C. 本次进入评估

**适用条件**：评估型输入 / 边界不清 / 触发判断问题 / 异常待判

**最小输出格式（3~4行）**：

```
[日期] 本次进入评估。
- 进入原因：[什么信号触发了评估需求]
- 当前判断：[评估结论，是否触发]
- 下一步：去决策页（ops-decision.html）做最终判断
- 状态影响：尚未进入 RUN-next，等待决策页结论
```

**示例**：

```
[日期] 本次进入评估。
- 进入原因：usable 池较上周下降 20%，不确定是否属于"明显异常"
- 当前判断：需要决策页判断是否命中触发条件
- 下一步：去 ops-decision.html 决策助手
- 状态影响：尚未触发，不进入 RUN-next
```

**不应写成**：
- "进入评估"写成"进入准备"（评估≠执行准备）
- 越过决策页直接给出触发结论
- 超过 10 行的详细分析

---

### D. 本次进入准备

**适用条件**：明确新数据 / 明显新信号 / 已接近或满足运行前条件

**最小输出格式（3~4行）**：

```
[日期] 本次进入准备。
- 准备原因：[什么信号表明需要准备]
- 当前状态：尚未正式进入 RUN-next，正在准备阶段
- 下一步：运营页 → 决策页 → 执行准备
- 状态影响：等待用户确认是否正式触发 RUN-next
```

**示例**：

```
[日期] 本次进入准备。
- 准备原因：曾刚连续 2 个月断档 + usable 池持续低于 5 条
- 当前状态：接近触发条件，需要决策页确认
- 下一步：ops-decision.html → ops-playbook.html 执行流程
- 状态影响：等待触发确认，不自动进入 RUN-next
```

**不应写成**：
- "进入准备"写成"RUN-next 已触发"（准备≠执行）
- 直接调用运行脚本
- 写完整的运行报告（那是 RUN-01 结束后的产出）

## 一句话模板速查

| 值班结果 | 一句话格式 |
|---------|----------|
| 本次无动作 | `[日期] 本次值班无动作。RUN-next 未触发，系统处于正常等待态。` |
| 本次仅记录 | `[日期] 本次仅记录：[口径/术语/回访内容]。状态无变化。` |
| 本次进入评估 | `[日期] 本次进入评估：[原因]。去决策页判断。` |
| 本次进入准备 | `[日期] 本次进入准备：[原因]。等待触发确认。` |

## 最小留痕字段（必须项）

每类值班结果的必须字段：

| 必须字段 | 无动作 | 仅记录 | 进入评估 | 进入准备 |
|---------|--------|--------|---------|---------|
| 日期 | ✅ | ✅ | ✅ | ✅ |
| 值班结果类型 | ✅ | ✅ | ✅ | ✅ |
| 主要内容/原因 | ❌ | ✅ | ✅ | ✅ |
| 状态影响 | ❌ | ✅ | ✅ | ✅ |
| 下一步动作 | ❌ | ❌ | ✅ | ✅ |

**最小留痕原则**：
- 无动作：只需日期 + 结果类型（一句话即可）
- 仅记录：日期 + 结果类型 + 记录内容（2~3行）
- 进入评估：日期 + 结果类型 + 原因 + 下一步（3~4行）
- 进入准备：日期 + 结果类型 + 原因 + 下一步（3~4行）

## 升级说明边界（普通留痕 vs 需要升级说明）

| 情形 | 类型 | 说明 |
|------|------|------|
| 快报"等待态正常"，无新信号 | 普通留痕 | 一句话即可，不需要升级 |
| 口径/术语修正 | 普通留痕 | 仅记录，不升级 |
| 有信号但不确定是否触发 | 普通留痕 | 去决策页判断，不升级 |
| usable 池明显异常下滑 | 升级说明 | 去决策页 + 可能需要通知相关人 |
| 曾刚连续断档 2 个月以上 | 升级说明 | 去决策页 + 考虑冻结条件 |
| 触发条件已明确命中 | 升级说明 | 去决策页 + 准备执行 |

**原则**：升级说明不是值班留痕，是"需要通知他人或触发决策"的信号。大多数值班只需要普通留痕，不需要升级说明。

---

## 当前最容易混淆的结果边界

| 混淆点 | 正解 |
|--------|------|
| 无动作写得太长 | 无动作只需一句话，不需要详细说明冻结状态 |
| 轻微信号写成进入评估 | 轻微信号属于"仅记录"，不进入评估 |
| 进入评估写成进入准备 | 评估是判断"要不要动"，准备是"准备开始动"，不同阶段 |
| 所有值班都写升级说明 | 大多数值班只需要普通留痕，不需要升级 |
| 仅记录写成触发结论 | 记录是"确认某事"，不是"判断某事" |
