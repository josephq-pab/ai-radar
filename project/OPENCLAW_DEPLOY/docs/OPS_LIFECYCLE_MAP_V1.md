## 背景

P3-35 已完成拍板回复吸收与决议回写规则，运营者现在知道"收到回复后怎么分类、怎么回写、停在哪"。

但仍存在一个问题：

> **整套体系的生命周期全貌没有被统一收口。** 运营者知道单点规则，但不知道这些规则在全流程中处于什么位置、状态之间如何流转、哪些转换必须人工判断。

当前误判：
- 等待态 ≠ 没有事情发生（等待态是系统的默认停点和默认起点）
- 进入评估 ≠ 必然升级（评估边界清晰时不需要升级）
- 进入准备 ≠ 已进入 RUN-next（准备是"准备开始"，触发是"正式开始"）
- 补材料 ≠ 否决（补完可重新升级）
- 暂不推进 ≠ 永久不做（下次有信号可重新升级）
- 拍板回复后 ≠ 自动流转（回复后需要人工判断下一步）

本功能包定义生命周期阶段、状态流转关系、不允许自动流转的边界。

---

## 生命周期阶段定义

| 阶段 | 代码 | 说明 | 是否终点 |
|------|------|------|---------|
| **等待态** | W | 默认起点 / 默认停点。无新信号、无动作、维持当前冻结边界。 | ✅ 是（默认终点）|
| **轻响应态** | L | 仅有口径澄清、术语查询、回访类输入。不进入评估，不需要升级。 | ❌ 否（中间状态）|
| **评估态** | A | 有边界问题、触发判断不清、异常待判、结构性输入出现。尚未准备，需要先评估。 | ❌ 否（中间状态）|
| **准备态** | P | 已接近下一步条件（触发条件接近命中/冻结条件接近满足）。尚未进入 RUN-next。 | ❌ 否（中间状态）|
| **升级态** | E | 需要人工拍板判断（边界不清/判断困难）或确认（准备请确认）。不是自动推进。 | ❌ 否（中间状态）|
| **回复吸收态** | R | 已收到升级回复。等待最小回写。不等于自动流转。 | ❌ 否（中间状态）|
| **回收停点** | S | 本次闭环。回到等待（W）、停在评估（A）、停在准备（P）、或停止（end）。 | ✅ 是（本次终点）|

## 状态流转关系

### 允许的流转路径

```
等待态(W)
    ↓ 新输入（口径/术语/回访）
轻响应态(L) → 回到等待态(W)（轻响应结束后）

等待态(W)
    ↓ 新输入（边界不清/触发待判）
评估态(A)
    ↓ 边界清晰，结论明确
回到等待态(W)（本次评估结束）

评估态(A)
    ↓ 需要人工判断
升级态(E)
    ↓ 收到回复
回复吸收态(R)
    ↓ 回写完成后
回到等待态(W)（本次升级闭环）

评估态(A)
    ↓ 触发条件接近命中/冻结条件接近满足
准备态(P)
    ↓ 需要人工确认
升级态(E)（准备请确认）
    ↓ 收到确认回复
回复吸收态(R)
    ↓ 回写完成后
停在准备态(P)（等待下一步）

等待态(W)
    ↓ 无新信号
等待态(W)（原地等待，默认停点）

任何状态
    ↓ 收到"暂不推进"回复
回到等待态(W)（维持当前）

任何状态
    ↓ 收到"不同意本次"回复
停在回收停点(S)（本次闭环）
```

### 不允许自动流转的边界

以下转换**不允许自动发生**，必须人工判断或确认：

| 转换路径 | 不允许原因 |
|---------|----------|
| 等待态 → 评估态 | 必须有新信号触发 |
| 评估态 → 升级态 | 必须明确满足升级条件（B/C级）|
| 评估态 → 准备态 | 必须明确满足准备条件 |
| 准备态 → 升级态（确认）| 必须获得人工确认 |
| 升级态 → 回复吸收态 | 必须收到实际回复 |
| 回复吸收态 → 等待态 | 必须完成回写 |
| 回复吸收态 → 继续准备 | 必须获得确认 |
| **任何状态 → RUN-next** | **RUN-next 必须由人工确认触发，V7 基线冻结** |

## 阶段与值班结果四分类对应

| 值班结果 | 主要对应阶段 | 说明 |
|---------|------------|------|
| 本次无动作 | W（等待态）| 没有新信号，维持等待 |
| 本次仅记录 | L（轻响应态）| 口径澄清/术语/回访，不进入评估 |
| 本次进入评估 | A（评估态）| 有边界问题/触发待判，但不必然升级 |
| 本次进入准备 | P（准备态）| 接近下一步，但不等于 RUN-next |

---

## 阶段与升级条件对应

| 升级层级 | 对应阶段 | 说明 |
|---------|---------|------|
| 不升级（默认）| W/L/A | 大多数值班不需要升级 |
| 评估请判断 | A（边界不清时）| 评估态 + 判断困难时才升级 |
| 准备请确认 | P | 准备态才需要"准备请确认" |

---

## 阶段与拍板回复类型对应

| 回复类型 | 回写后停在 | 说明 |
|---------|----------|------|
| 同意推进 | A（继续评估）或 P（继续准备）| 进入下一步，但不等于 RUN-next |
| 暂不推进 | W | 本次闭环，回到等待态 |
| 需补材料 | A | 补完后重新升级 |
| 不同意本次 | S | 本次闭环，停在回收停点 |

## 生命周期全路径速查

```
日常值班开始
    ↓
① 日常值班最小巡检（OPS_DAILY_ROUTINE_V1.md）
    → 判断是否有新信号
    ↓ 无新信号
等待态(W) = 本次无动作 → 值班结束
    ↓ 有新信号
② 值班结果判断（OPS_SHIFT_OUTPUT_V1.md）
    → 判断属于哪类结果
    ↓
    ├─ 仅记录 → 轻响应态(L) → 值班结束
    ├─ 进入评估 → 评估态(A)
    │   ↓ 边界不清/判断困难
    │   ③ 升级说明（OPS_ESCALATION_PACK_V1.md）
    │       → 发起升级
    │       ↓ 收到回复
    │   ④ 回复吸收（OPS_DECISION_REPLY_V1.md）
    │       → 回写 + 停点
    │       ↓
    │       ├─ 同意 → 继续评估 或 进入准备
    │       ├─ 暂不 → 等待态(W)
    │       ├─ 补材料 → 评估态(A)（补完重新升级）
    │       └─ 不同意 → 本次闭环
    │
    └─ 进入准备 → 准备态(P)
        ↓ 需要确认
        ③ 升级说明（准备请确认）
            → 发起升级
            ↓ 收到确认回复
        ④ 回复吸收
            → 回写 + 停点
            ↓
            ├─ 确认 → 准备态(P)（等待RUN-next确认）
            └─ 不同意 → 本次闭环
```

## 常见误判关系澄清

| 误判 | 正确理解 |
|------|---------|
| 等待态 = 没有事情发生 | 等待态是默认停点和默认起点，系统大多数时候停在等待态 |
| 进入评估 = 一定要升级 | 评估边界清晰、结论明确时，不需要升级 |
| 进入准备 = 已经运行 | 准备是"准备开始"，RUN-next 是"正式开始"，完全不同阶段 |
| 补材料 = 否决 | 需补材料不是否决，补完材料后可重新升级 |
| 暂不推进 = 永久不做 | 暂不推进不是永久否决，下次有明确信号可重新升级 |
| 不同意本次 = 永久否决 | 本次不同意不是永久否决，有新信号可重新升级 |
| 收到回复 = 自动进入下一状态 | 回复后必须先分类，再决定停在哪，不会自动流转 |
| 升级后 = RUN-next 已触发 | 升级只是请求判断/确认，不等于 RUN-next 已触发 |
| 阅读路径 = 生命周期路径 | 阅读路径是理解系统的路径，生命周期路径是系统实际运转的路径，二者不同 |
