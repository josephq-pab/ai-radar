## 一、治理对象总览

全站治理对象共 3 层：

```
┌─────────────────────────────────────────────────────────┐
│ 第一层：全局真源层（canonical baseline）                   │
│   全站只有一份，不得随意改动                               │
│   改了这一层，其他层必须同步                              │
├─────────────────────────────────────────────────────────┤
│ 第二层：展示与规则定义层                                  │
│   以真源为依据，对外展示规则                              │
│   改了真源层，这里必须查是否需要同步                       │
├─────────────────────────────────────────────────────────┤
│ 第三层：训练与辅助说明层                                  │
│   以真源和展示层为依据，做训练/演练/辅助说明               │
│   这一层改了，不应反向改动真源层或展示层                   │
└─────────────────────────────────────────────────────────┘
```

## 二、各层具体对象

### 第一层：全局真源层（canonical baseline）

| 对象 | 治理角色 | 说明 |
|------|---------|------|
| STAGE2_SINGLE_CHAIN_STATUS.md | 全站状态主线唯一真源 | 所有状态演进、阶段定义、流转规则的权威来源 |
| V7 基线 | 当前冻结基线 | 脚本/配置/基线均冻结，不主动改动 |
| analyst_sources.json | 分析师来源配置 | 结构性来源，当前为影子模式，不主动改动 |

**为什么这层是全局真源：**
- 这三个对象是整个体系的"根"
- 其他所有页面和文档都从这三个对象派生
- 改了这三个，其他层必须全部检查是否需要同步

## 三、全局真源层级优先规则

当不同层级的对象对同一事项描述不一致时，以更高层级为准：

```
STAGE2 > V7基线 > analyst_sources.json > 各HTML页面 > 各文档包
```

**实际应用：**
- 若 radar-home.html 的状态主线句与 STAGE2 不一致 → 以 STAGE2 为准，修改 radar-home
- 若 OPS_TRAINING_PACK 的某术语定义与 ops-glossary.html 不一致 → 以 ops-glossary 为准，修改训练包
- 若 ops-decision.html 的决策规则与 STAGE2 不一致 → 以 STAGE2 为准，修改 ops-decision

## 四、哪些对象是"全局硬约束"

以下词句/模块在全站有且只有一种写法，改了必须全站同步：

| 硬约束对象 | 标准写法 | 全站统一的重要性 |
|-----------|---------|---------------|
| 状态主线句 | "对公 AI 雷达站 · V7 基线冻结 · 十页入口可用" | 高（使用者第一印象）|
| 七阶段写法 | W/L/A/P/E/R/S（Waiting/Light response/Assessment/Preparation/Evaluation/Review/Stop）| 极高（全站统一才能协作）|
| 生命周期禁止自动流转 | "准备态(P) ≠ RUN-next 已触发" | 极高（误判会跳过关键步骤）|
| 来源候选≠解冻 | "来源候选 ≠ 解冻" | 高（违反会破坏口径一致性）|
| 补材料≠否决 | "补材料 = 信息不够，补完可继续" | 高（误判会错误放弃有效输入）|
| 暂不推进≠否决 | "暂不推进 = 维持等待态(W)，条件变可再提" | 高（误判会错误放弃有效输入）|
| 快报≠规则学习材料 | "快读 = 1~2句状态，不是完整分析" | 中（不影响协作，只影响使用体验）|
| 证据页≠默认入口 | "只在解释/补材料/成立依据时进入" | 高（误用会倒置判断路径）|
| 决策页≠默认入口 | "只在边界不清/触发待判/异常待判时进入" | 高（误用会造成决策资源浪费）|

## 五、治理边界：哪些事本轮/本层不做

### 本轮（P3-41）不做

- 不改脚本/analyst_sources.json/V7基线
- 不新增入口页
- 不进入RUN-02
- 不做第二链路解冻
- 不做后台/UI重构/数据库

### 长期治理边界

- 第一层（真源层）不做运行时动态修改
- 第二层（展示层）不做自动化内容生成
- 第三层（训练层）不允许反向改动第一/二层

## 六、后续改动时的治理检查顺序

```
第1步：判断改动涉及哪个层级
  ↓
第2步：查 OPS_CHANGE_IMPACT_MATRIX_V1.md，看必须同步哪些对象
  ↓
第3步：先改真源层，再改展示层，最后检查训练层
  ↓
第4步：改完后对照 OPS_TRACEABILITY_MATRIX_V1.md 验证
  ↓
第5步：更新 CHANGE_CONTROL.md + REVIEW_LOG.md
  ↓
第6步：若涉及页面改动，运行最小验证清单
```
