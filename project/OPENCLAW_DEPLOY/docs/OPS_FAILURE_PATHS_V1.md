## 二、失败路径详解

### FP-01：只改 banner 不同步真源

**对应 MT：** MT-01 / MT-02

**错误描述：**
在展示层（如 radar-home banner）改了 S0 句，但没有更新 STAGE2 真源层。后续真源更新时，展示层会被覆盖，改动丢失。

**错误顺序：**
```
1. 在 radar-home banner 改 S0 句
2. 未更新 STAGE2
3. 后续 STAGE2 被同步，展示层被覆盖
4. 改动丢失，出现口径不一致
```

**会破坏什么：**
- 真源层与展示层不一致
- 后续同步时改动被覆盖
- 全站口径不一致

**最早在哪发现：**
- 改动后 grep 全站，发现 STAGE2 与页面内容不一致

**止损方式：**
1. 立即停止展示层改动
2. 先更新 STAGE2 真源
3. 再从真源同步到所有展示层页面

**预防：**
改 S0/S1 句前，先确认 OPS_MODULE_DEPENDENCY_MAP_V1.md，查"上游真源"字段

---

### FP-02：只改一个镜像页不同步其他镜像

**对应 MT：** MT-01 / MT-05

**错误描述：**
S0/S1 句在多个页面有镜像副本（如 ops-brief 和 ops-decision 都有同一句），只改了其中一个，未同步其他。

**错误顺序：**
```
1. 在 ops-brief 改了某 S1 句
2. 未检查 ops-decision / single-chain-ops 的对应镜像
3. 用户在不同页面看到不一致的内容
4. 出现"口径冲突"投诉
```

**会破坏什么：**
- 全站口径一致性
- 用户信任（不同页面说法不一）
- S1 同步机制失效

**最早在哪发现：**
- 改后 grep 全站，发现其他页面镜像未更新

**止损方式：**
1. 确认改动后"正确的版本"
2. 把所有镜像页面同步为正确版本
3. 补全版本标注

**预防：**
改 S1 句前，先 grep 全站列出所有出现位置

---

### FP-03：把 S0 句的含义改漂移了

**对应 MT：** MT-01 / MT-02 / MT-03

**错误描述：**
改 S0 句时，把"措辞调整"做成了"含义改变"，导致原句的核心约束被破坏。

**典型错误：**
- "冻结≠没做" → "冻结≠无进展"（"没做"已是最强澄清，换"无进展"削弱了约束）
- "准备≠RUN-next已开启" → "准备好就可以跑"（去掉了"≠"的核心约束）
- "第二链路冻结（结构性来源缺失，不自动解冻）" → "第二链路暂停（待解决）"（"暂停"暗示会恢复）

**错误顺序：**
```
1. 觉得原句措辞不够通顺
2. 改了措辞，但改变了核心约束
3. 全站同步到所有镜像
4. 用户按照新含义操作，触发误判
```

**会破坏什么：**
- S0 硬约束机制失效
- 全站高频误判澄清能力丧失
- 引发新的误判风险

**最早在哪发现：**
- 对照 OPS_SENTENCE_REGISTRY_V1.md 中的 canonical 版本，发现新版本语义已漂移

**止损方式：**
1. 立即回退所有受影响页面
2. 恢复 canonical 版本
3. 重新评估改动必要性

**预防：**
改 S0 句前，必须对照 canonical 版本，确认"改完后的含义与原意相同"

---

### FP-04：把训练句改进判断层

**对应 MT：** MT-09 / MT-11

**错误描述：**
在训练提示层（如 P3-40 训练提示）加了实质性判断内容，导致训练提示变成了隐性判断依据。

**典型错误：**
- 在"30秒学会什么"训练提示中，加了"当 X 时必须升级"的实质性判断
- 在误判澄清说明中，把澄清句写成了判断规则

**会破坏什么：**
- 训练层与判断层职责混淆
- 用户按训练提示做判断，引发误判
- 页面角色边界混乱

**止损方式：**
1. 回退训练层改动
2. 确认改动内容应该放在训练层还是判断层
3. 如确实需要，加判断层（single-chain-ops / ops-decision），训练层只保留说明

---

### FP-05：把"来源候选"写成"解冻"

**对应 MT：** MT-04

**错误描述：**
把"来源候选≠解冻"的澄清句，改成了"来源候选=解冻"或模糊化了"≠"。

**典型错误：**
- "来源候选≠解冻（必须经过评估才能解冻）" → "已有来源候选，可解冻"
- 把澄清句删掉

**会破坏什么：**
- 第二链路防误用机制失效
- 用户误以为"有候选=可以解冻"
- 违反 V7 冻结体系

**止损方式：**
1. 立即恢复 canonical 版本
2. 确认为什么用户会有"候选=解冻"的误解
3. 如需加说明，在第二链路说明中补充，不改动"来源候选≠解冻"句

---

### FP-06：只改镜像页不更新 P3 版本

**对应 MT：** MT-01~MT-16（全类型）

**错误描述：**
改了页面内容，但忘记更新 footer 的 P3 版本标注，导致后续维护者不知道哪个页面被改过。

**会破坏什么：**
- 维护者无法追溯变更历史
- 版本不一致引发额外排查成本

**止损方式：**
补齐所有被改动页面的 footer 版本标注

**预防：**
把"版本标注跟着改动走"作为每次改动的强制环节

---

### FP-07：改完 S1 句但不查 OPS_SENTENCE_DIFF_MATRIX

**对应 MT：** MT-05 / MT-06

**错误描述：**
改了某 S1 句在 A 页面的表达，但没有查 OPS_SENTENCE_DIFF_MATRIX_V1.md 确认"这个句子的其他版本是否需要同步"。

**会破坏什么：**
- OPS_SENTENCE_DIFF_MATRIX_V1.md 中记录为"已统一"的项重新出现差异
- 后续维护者查阅矩阵时发现不一致

**止损方式：**
1. 查 OPS_SENTENCE_DIFF_MATRIX_V1.md
2. 确认该句差异状态（允许差异 / 需同步 / 已统一）
3. 如需同步，同步所有页面

---

### FP-08：深夜改 No-touch Zone

**对应 MT：** MT-01 / MT-02 / MT-03 / MT-04 / MT-15

**错误描述：**
在深夜（00:00~06:00）执行 No-touch Zone 的 S0 句改动，因为疲劳和时间压力，没有完成完整的 FC-1~FC-7 回归，导致遗漏。

**会破坏什么：**
- 改动引入漂移但未被发现
- 深夜无人验收，改动后果被放大

**止损方式：**
1. 立即停止继续改动
2. 完成 FC-1~FC-7 全部回归步骤
3. 白天重新人工验收

**预防：**
建立"深夜 No-touch Zone 禁止改动"的强制规则

---

### FP-09：把近重复当完全重复压缩

**对应 MT：** MT-16

**错误描述：**
看到两处"看起来差不多"的句子，误判为"完全重复"并压缩掉其中一处，实际上它们是"角色性重复"（有意设计），压缩后丢失了页面的角色差异化表达。

**典型错误：**
- 把 ops-brief 的"准备≠执行"和 canonical "准备≠RUN-next已开启"误判为完全重复
- 把首页的"冻结≠没做/阻断≠放弃/不做≠忘了做"三连句和 ops-decision 的展开段落误判为重复

**会破坏什么：**
- 页面角色差异化设计丧失
- 用户失去快读版和详细版的区分
- 引入新的理解困难

**止损方式：**
1. 立即恢复被删除的内容
2. 查 OPS_MODULE_REDUNDANCY_REVIEW_V1.md，确认重复类型
3. 如是角色性重复，明确记录"保留，各自独立维护"

**预防：**
改前查 OPS_MODULE_REDUNDANCY_REVIEW_V1.md 的"近重复/角色性重复"分类

---

### FP-10：改了模块标题但引用处未更新

**对应 MT：** MT-08

**错误描述：**
改了某模块的标题（如"当前运营总览" → "当前运营状态总览"），但其他页面引用该标题时仍用旧名称。

**会破坏什么：**
- 用户按路径走到"首页 → 当前运营总览"发现没有这个模块
- 引用链断裂

**止损方式：**
1. 全站 grep 搜索旧标题名称
2. 同步更新所有引用处（或明确记录引用处保留旧名称，待后续统一）

**预防：**
改模块标题前，grep 全站确认无其他引用

---

### FP-11：改完真源但忘记同步展示层

**对应 MT：** MT-01 / MT-03 / MT-04 / MT-12 / MT-15

**错误描述：**
更新了 STAGE2/OPS_LIFECYCLE_MAP 等真源文档，但忘记同步前端页面的对应内容。

**会破坏什么：**
- 真源层与展示层脱节
- 后续展示层同步时覆盖改动

**止损方式：**
1. 确认真源文档的正确版本
2. 逐页检查展示层页面，确认哪些已同步、哪些未同步
3. 同步所有未更新的展示层页面

**预防：**
改真源前，grep 全站列出所有引用该真源的展示页面，改完后强制同步

---

### FP-12：改完即结束，不做 QC 回归

**对应 MT：** MT-07 / MT-09 / MT-10（全类型）

**错误描述：**
改完内容就认为完成了，没有做任何 QC 回归检查，结果改动引入明显错误。

**止损方式：**
1. 打开改动后的页面
2. 确认加载正常 / 上下文通顺
3. 如发现问题，回退改动

**预防：**
把 QC-1~QC-2 作为每次改动的最低要求，强制执行

## 三、失败路径归并：6 类失败母型

### 母型 A：真源-展示不同步

**涵盖 FP-01 / FP-11**
**本质：** 改了真源但展示层没更新，或反之
**最早预警：** 改动后 grep 真源和展示层内容不一致
**止损核心：** 先定"正确版本"，再全量同步

### 母型 B：镜像副本遗漏

**涵盖 FP-02 / FP-07**
**本质：** 多个镜像副本中只改了其中一个
**最早预警：** grep 全站发现其他位置内容未更新
**止损核心：** 改前列出所有镜像位置，改后逐个确认

### 母型 C：S0 句含义漂移

**涵盖 FP-03 / FP-05**
**本质：** 措辞改了但核心约束变了
**最早预警：** 对照 canonical 版本发现语义已变
**止损核心：** 立即回退，恢复 canonical

### 母型 D：角色层混淆

**涵盖 FP-04 / FP-09**
**本质：** 把 A 层的内容放进了 B 层
**最早预警：** 改动后发现"这个内容不应该在这里"
**止损核心：** 回退，重新判断内容归属层

### 母型 E：深夜高风险操作

**涵盖 FP-08**
**本质：** 深夜执行 No-touch 改动，疲劳导致遗漏
**最早预警：** 时间戳显示深夜 + 改动涉及 S0/P0
**止损核心：** 停止，补 FC，人工验收

### 母型 F：版本标注遗漏

**涵盖 FP-06 / FP-12**
**本质：** 改动完了但没记录
**最早预警：** 后续查看 footer 发现版本未更新
**止损核心：** 补齐版本标注，把版本更新作为强制环节
