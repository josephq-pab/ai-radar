## 一、常见改坏模式清单

### BP-01：只改一个页面不同步其他镜像

**模式描述：**
某个 S0/S1 句在一个页面被改了，但其他页面的镜像副本没有同步更新，导致全站口径不一致。

**典型场景：**
- 改了 ops-brief 的 V7 冻结说明，但 radar-home 的 banner 没有同步
- 改了 single-chain-ops 的升级说明，但 ops-decision 的对应句没有同步
- 改了首页 footer 的状态句，但 ops-brief banner 没有同步

**预警信号：**
- 改动涉及 S0/S1 句
- 改动涉及 ≥2 页面的镜像副本
- 改完后没有做全站镜像同步检查

**如何尽早发现：**
- 改前查 OPS_MODULE_DEPENDENCY_MAP，确认镜像副本数量
- 改后 grep 全站搜索该句，确认所有位置是否一致

**止损步骤（不改大范围的前提下）：**
1. 确认改动后最新的"正确版本"是什么
2. 把所有遗漏的镜像页面同步为最新版本
3. 做全站 grep 搜索，确认无其他遗漏
4. 记录本次失误，下次改 S0/S1 前增加"镜像同步"检查项

### BP-02：改 S0 句含义但只改了一个页面的措辞

**模式描述：**
改 S0 句时，把"措辞调整"做成了"含义改变"，导致全站标准句含义漂移。

**典型场景：**
- 把"第二链路冻结（结构性来源缺失，不自动解冻）"改成了"第二链路暂停（待解决）"——改变了"不自动解冻"的核心约束
- 把"升级是请求判断，不是汇报进展"改成了"升级是汇报进展"——完全反转含义
- 把"准备≠RUN-next已开启"改成了"准备好就可以跑"——去掉了"≠"的核心约束

**预警信号：**
- 改动涉及 S0 句，但改动时没有对照 OPS_SENTENCE_REGISTRY_V1.md 中的 canonical 版本
- 改动的理由是"措辞更通顺"而不是"canonical 版本本身有误"
- 改动后没有在 S0 句 registry 中验证新版本是否仍保持 canonical 含义

**如何尽早发现：**
- 改前对照 OPS_SENTENCE_REGISTRY_V1.md 中的 canonical 版本
- 改后问：这个改动改变了原句的核心约束吗？
- 改后查 OPS_SENTENCE_DIFF_MATRIX，确认新版本在所有页面的含义是否一致

**止损步骤：**
1. 立即回退所有受影响页面的改动
2. 重新对照 canonical 版本，确认改动目标
3. 如果 canonical 版本本身无误，恢复 canonical 版本
4. 如果 canonical 版本确实需要更新，走正式变更流程（CC-xxx）

---

### BP-03：改模块标题导致引用链断裂

**模式描述：**
改了某个模块的标题，但其他页面引用该标题时仍用旧名称，导致"页面A说要看模块X，但模块X在新版本里叫模块Y"，用户困惑。

**典型场景：**
- 改了 home-M01 的标题，但 ops-routes 的路径说明仍引用"首页-模块M01"
- 改了 ops-registry-M04 的标题，但 radar-home 的治理说明引用旧名称

**预警信号：**
- 改动涉及模块标题
- 改前没有做全站 grep 搜索该标题的引用

**如何尽早发现：**
- 改模块标题之前，grep 全站搜索该标题/标签名称
- 确认是否有其他页面引用该标题

**止损步骤：**
1. 全站 grep 搜索旧标题名称
2. 同步更新所有引用处（可以一起改，也可以选择不改——不改引用也是一种选择）
3. 如果选择保留旧引用，明确记录"此处引用的是旧标题名称，待后续统一"

---

### BP-04：改完 S1 句但忘记更新版本标注

**模式描述：**
改了某个页面的内容，但忘记更新页面底部的 P3 版本标注，导致维护者无法追溯变更历史。

**典型场景：**
- 改了 ops-brief 的一个模块，但没有把版本从 P3-43 更新到 P3-44
- 改了多个页面的内容，但只更新了部分页面的版本标注

**预警信号：**
- 改动涉及任何页面内容
- 改完后没有统一的"版本标注更新"环节

**如何尽早发现：**
- 养成习惯：任何页面内容改动，必须更新该页 P3 版本标注
- 改完后查 OPS_MODULE_INVENTORY_V1.md 中对应模块的版本字段

**止损步骤：**
1. 确认所有被改动页面的最新 P3 版本
2. 补齐所有遗漏的版本标注
3. 记录本次失误，下次建立"版本标注更新"检查项

---

### BP-05：夜间改高风险内容

**模式描述：**
在夜间执行 No-touch Zone 的改动，因为疲劳或时间压力，没有完成完整的 Full Check 回归，导致遗漏。

**典型场景：**
- 深夜修改 V7 冻结相关句子，只改了一个页面就结束了
- 深夜改生命周期核心句，没有同步所有页面

**预警信号：**
- 改动时间在深夜（00:00~06:00）
- 改动涉及 No-touch Zone
- 没有完成 FC-1 ~ FC-7 全部步骤

**如何尽早发现：**
- 自我判断：现在是深夜吗？改动涉及 No-touch Zone 吗？
- 如果两个答案都是"是"，建议白天再做

**止损步骤：**
1. 立即停止继续改动
2. 完成 FC-1 ~ FC-7 的完整回归步骤（如已完成，跳过）
3. 人工打开所有涉及页面确认
4. 记录本次深夜改动，下次建立"深夜不改 No-touch Zone"规则

---

### BP-06：把"近重复"当"完全重复"压缩

**模式描述：**
看到两处"看起来差不多"的句子，误判为"完全重复"并压缩掉其中一处，实际上它们是"近重复"（有意设计），压缩后丢失了页面的角色差异化表达。

**典型场景：**
- 把 ops-brief 的快读版"准备≠执行"和 canonical "准备≠RUN-next已开启"误判为完全重复，删除其中一个
- 把首页的"冻结≠没做/阻断≠放弃/不做≠忘了做"三连句和 ops-decision 的"冻结不是停摆"展开段落误判为重复，删掉三连句

**预警信号：**
- 改动是"压缩/合并/删除"类操作
- 改前没有区分"完全重复"和"近重复/角色性重复"
- 改前没有查 OPS_MODULE_REDUNDANCY_REVIEW_V1.md

**如何尽早发现：**
- 改前查 OPS_MODULE_REDUNDANCY_REVIEW_V1.md，确认该内容属于哪类重复
- 确认是"完全重复"还是"角色性重复"
- 如果是角色性重复，不压缩

**止损步骤：**
1. 如果压缩已发生，立即恢复被删除的内容
2. 重新确认重复类型
3. 如果确认是角色性重复，明确记录"保留，各自独立维护"

---

### BP-07：改完真源层但忘记同步展示层

**模式描述：**
更新了 STAGE2/OPS_COPY_BASELINE 等真源文档，但忘记同步前端页面的对应内容，导致真源和展示不一致。

**典型场景：**
- 更新了 STAGE2_SINGLE_CHAIN_STATUS.md 的状态描述，但 radar-home 的状态横幅没有更新
- 更新了 OPS_LIFECYCLE_MAP 的生命周期定义，但 ops-glossary 没有更新

**预警信号：**
- 改动涉及真源文档（STAGE2/OPS_COPY_BASELINE/OPS_LIFECYCLE_MAP）
- 没有"真源→展示层"同步的强制环节

**如何尽早发现：**
- 改真源前先 grep 全站，列出所有引用该真源的展示页面
- 改完真源后，检查所有展示页面是否已同步

**止损步骤：**
1. 确认真源文档的最新正确版本
2. 逐页检查展示层页面，确认哪些已同步、哪些未同步
3. 同步所有未更新的展示层页面
4. 记录本次失误，下次建立"真源改动→展示层同步"检查项

---

### BP-08：改动影响 P0 模块但只改了一个页面

**模式描述：**
改动涉及 P0 模块（如 ops-M09 次轮触发规则摘要），但只改了一个页面，没有做全站 P0 同步，引发高风险漂移。

**典型场景：**
- 改了 single-chain-ops 的次轮触发规则，但 ops-brief 和 ops-decision 的对应引用没有同步
- 改了 registry-M07 冲突解决规则，但 radar-home 的真源引用没有更新

**预警信号：**
- 改动涉及 P0 模块
- 改前没有查 OPS_MODULE_PRIORITY_MATRIX_V1.md

**如何尽早发现：**
- 改前查 OPS_MODULE_PRIORITY_MATRIX_V1.md，确认是否 P0
- 如果是 P0，必须全站同步

**止损步骤：**
1. 立即确认 P0 模块的所有出现位置
2. 同步所有出现位置为最新版本
3. 完成 FC-1 ~ FC-7 全部回归步骤
4. 记录本次失误

## 二、止损决策树

```
发现改坏了？

↓

是 S0/S1 句改动导致的吗？
  → 是：回退所有受影响页面 → 重新对 canonical 版本 → 再评估

是模块标题/引用链问题吗？
  → 是：grep 全站搜索旧名称 → 同步所有引用 → 记录

是真源层/展示层不同步吗？
  → 是：先更新真源 → 再同步展示层 → 确认一致性

是版本标注遗漏吗？
  → 是：补齐所有版本标注 → 记录

是深夜改 No-touch 吗？
  → 是：立即停止 → 完成 FC 步骤 → 白天重新验收

以上都不是，但页面确实有问题？
  → 是：回退改动 → 重新理解上下文 → 再评估是否值得改
```

## 三、不扩改止损原则

**止损时最重要的原则：不扩改。**

改坏了之后的第一反应应该是"回退"，而不是"继续改其他页面来凑"。

| 扩改风险 | 正确做法 |
|---------|---------|
| 改坏 A → 继续改 B/C/D 来同步 | ❌ 扩改引入新风险 | 先回退 A，再系统性地查 B/C/D |
| 改坏后时间紧，干脆不记录 | ❌ 后续维护者不知道 | 必须记录本次失误 |
| 改坏后悄悄改回去不告诉任何人 | ❌ 问题被掩盖 | 记录失误，更新 CHECKLIST |

## 四、预防性检查清单（在改动前）

**改 S0/S1 句之前：**
```
[ ] 对照 OPS_SENTENCE_REGISTRY_V1.md，确认 canonical 版本
[ ] grep 全站搜索该句的所有出现位置
[ ] 确认所有出现位置是否需要同步
[ ] 准备好同步所有镜像
```

**改模块标题之前：**
```
[ ] grep 全站搜索该标题，确认是否有其他引用
[ ] 确认引用处是否需要一起更新
```

**改真源文档之前：**
```
[ ] grep 全站，列出所有引用该真源的展示页面
[ ] 准备好展示层同步方案
```

**改 P0 模块之前：**
```
[ ] 查 OPS_MODULE_PRIORITY_MATRIX_V1.md 确认 P0
[ ] 列出所有出现位置（通常 ≥3）
[ ] 确认全站同步方案
```

**深夜改任何内容之前：**
```
[ ] 确认是否涉及 No-touch Zone
[ ] 如果是 No-touch Zone，建议白天再做
[ ] 如果必须深夜改，完成 FC-1~FC-7 全部步骤
```
