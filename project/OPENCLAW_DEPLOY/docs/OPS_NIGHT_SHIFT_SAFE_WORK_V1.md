# OPS_NIGHT_SHIFT_SAFE_WORK_V1.md — 夜间适配与白天确认边界

> 文档版本：v1.0（P3-45）
> 创建时间：2026-04-21
> 对应阶段：Phase 3 — P3-45 全站黄金路径、失败路径与维护配方样本库功能包第一期
> 目的：明确哪些改动很适合夜里做、哪些应留到白天、哪些只分析不实施，让夜间维护有据可依

---

## 一、夜间维护的基本原则

**夜间（00:00~06:00）核心约束：**
- 精力有限，复杂回归容易遗漏
- 无人实时验收，错误可能被放大
- No-touch Zone 改动在夜间禁止
- 大范围（≥3 页联动）改动不建议夜间

**夜间适合做的事：**
- Safe Zone 低风险改动
- 单一页面内的 S2/S3 句微调
- 文档更新（不涉及页面展示）
- 版本标注更新
- 纯分析不实施

**夜间不适合做的事：**
- No-touch Zone 任何改动
- S0 句改动
- ≥3 页联动的 S1 句改动
- P0 模块改动
- glossary M01~M07 改动
- registry-M04 / M07 改动
- cleanup 压缩类改动

## 二、夜间可直接执行的改动类型

### 2.1 可夜间执行的配方（🟢 Safe Zone + 🟡 Controlled 部分）

| 配方 | 改动类型 | 夜间可行原因 | 最低回归要求 |
|------|---------|------------|------------|
| R-06 | 改本页面内 S1 句（≤2页联动） | 风险可控 | QC-1~QC-2 |
| R-07 | 改 S2/S3 句 | 不影响判断逻辑 | QC-1 |
| R-08 | 改模块标题（无引用链断裂） | 局部改动 | QC-1~QC-2 |
| R-09 | 改训练提示句 | 训练层不参与判断 | QC-1 |
| R-10 | 改路径引导句 | 局部改动 | QC-1~QC-2 |
| R-13 | 改 evidence 说明句（≤2页） | 局部改动 | QC-1~QC-2 |
| R-14 | 改配置描述 | 配置层不参与判断 | QC-1 |

### 2.2 夜间可执行但建议白天做的配方（⚠️ Controlled 部分）

以下配方夜间可以执行，但建议优先选白天：

| 配方 | 改动类型 | 建议白天原因 | 如必须夜间 |
|------|---------|------------|-----------|
| R-05 | 改跨页 S1 句（≥3页联动） | 影响面大，夜间容易遗漏 | 完成 FC-1~FC-7，白天重新验收 |
| R-11 | 改边界对照说明（≥2页联动） | 影响边界判断 | 完成 QC-3，白天重新验收 |
| R-12 | 改 registry 归属句 | registry-M04/M07 影响真源裁判 | 禁止夜间（建议白天） |
| R-16 | cleanup 压缩 | 压缩后语义容易丢失 | 禁止夜间（建议白天） |

## 三、深夜（00:00~06:00）禁止执行的改动

以下改动在深夜 00:00~06:00 绝对禁止：

### 🔴 禁止深夜执行的改动清单

| # | 禁止深夜执行的改动 | 原因 |
|---|------------------|------|
| 1 | 改任何 S0 句（MT-01~04） | S0 硬约束，全站标准，漂移风险极高 |
| 2 | 改 glossary M01~M07（W/L/A/P/E/R/S 定义） | 全站生命周期标准句，影响 ≥5 页 |
| 3 | 改 registry-M04（十页真源归属总表） | 真源裁判基础，变动影响大 |
| 4 | 改 registry-M07（冲突解决规则） | 全站真源冲突裁判句 |
| 5 | 改任何 banner 中的状态句 | banner 是全站状态主线的门面 |
| 6 | 改第二链路冻结句（MT-04） | V7 冻结体系核心，禁止漂移 |
| 7 | 改 V7 基线相关句 | V7 基线冻结，禁止任何形式改动 |
| 8 | 改 P0 模块（MT-01~04 覆盖之外的 P0） | P0 模块是高依赖核心模块 |
| 9 | 任何涉及 ≥5 页的大范围改动 | 深夜大范围影响 = 高风险失控 |
| 10 | 任何 cleanup 压缩类改动 | 压缩后语义容易丢失，夜间难以发现 |

## 四、深夜应"只分析不实施"的场景

以下场景，深夜只做分析，不实施：

### 4.1 S0/S1 句改动请求

**场景：** 用户提出要改某个 S0 句或跨页 S1 句

**深夜做法：**
```
[ ] 记录改动请求（copy 到分析文档）
[ ] 查 OPS_SENTENCE_REGISTRY_V1.md，确认该句 canonical 版本
[ ] 列出所有出现位置
[ ] 评估影响范围
[ ] 给出"建议白天实施"的理由
[ ] 不实施，等白天
```

### 4.2 生命周期/W/L/A/P/E/R/S 定义改动请求

**场景：** 用户提出要改某个阶段定义

**深夜做法：**
```
[ ] 记录改动请求
[ ] 查 OPS_LIFECYCLE_MAP_V1.md，确认真源版本
[ ] 列出所有出现位置（通常 ≥5）
[ ] 给出"建议白天实施+完整 FC 回归"的理由
[ ] 不实施，等白天
```

### 4.3 cleanup 压缩请求

**场景：** 用户提出要合并某两处"看起来重复"的段落

**深夜做法：**
```
[ ] 记录改动请求
[ ] 查 OPS_MODULE_REDUNDANCY_REVIEW_V1.md，确认重复类型
[ ] 如为"角色性重复"，告知"禁止压缩，维持现状"
[ ] 如为"近重复"，评估风险，建议白天做
[ ] 不实施，等白天
```

### 4.4 真源层更新请求

**场景：** 用户提出要更新 STAGE2 / OPS_LIFECYCLE_MAP 等真源文档

**深夜做法：**
```
[ ] 记录改动请求
[ ] 列出所有引用该真源的展示页面
[ ] 评估展示层同步工作量
[ ] 给出"建议白天做，展示层同步后验收"的理由
[ ] 不实施，等白天
```

## 五、夜间任务自检表

每次夜间准备执行改动前，必须先过一遍：

```
夜间任务自检（00:00~06:00）：

[ ] 改动属于 R-06~R-10,R-13~R-14（可夜间）吗？
    → 是：继续自检
    → 否：停止，等白天

[ ] 改动涉及 S0 句吗？
    → 是：停止，等白天

[ ] 改动涉及 glossary M01~M07 吗？
    → 是：停止，等白天

[ ] 改动涉及 registry-M04/M07 吗？
    → 是：停止，等白天

[ ] 改动影响 ≥3 页吗？
    → 是：升级到 FC-1~FC-7，完成后白天重新验收

[ ] 当前是深夜（00:00~06:00）吗？
    → 是，且涉及 No-touch/No-touch-adjacent → 停止，等白天
    → 否：可以继续

[ ] 有明确的改动理由和预期结果吗？
    → 否：停止，不值得改

以上全部通过：按配方执行 QC-1~QC-2
```

## 六、夜间任务完成后的最低要求

夜间改动完成后，必须确认：

```
[1] 改动页面已自查（QC-1）
[2] 相关引用页已确认（QC-2）
[3] footer P3 版本已更新
[4] 如涉及 S1 句，OPS_SENTENCE_DIFF_MATRIX_V1.md 已记录（如需）
[5] 改动内容记录到 maintenance log（如有）
```

## 七、夜间适合新建的文档类型

夜间只适合新建/更新纯文档，不影响展示层：

| 夜间可新建/更新的文档 | 说明 |
|---------------------|------|
| OPS 治理文档（如 OPS_*.md 新建章节） | 纯文档，不影响页面 |
| REVIEW_LOG / CHANGE_CONTROL | 跟踪文档，纯记录 |
| memory/ 复盘文件 | 纯记录 |
| 文档版本标注更新 | 纯元信息 |

## 八、夜间任务边界总结

```
✅ 可夜间实施：
  R-06~R-10（单一页面 Safe Zone / ≤2页 Controlled）
  R-14（配置描述）
  文档新建/更新
  版本标注更新

⚠️ 需完整 FC+白天重新验收：
  R-05（≥3页联动 S1 句）
  R-11（≥2页联动边界对照）

❌ 禁止夜间：
  R-01~R-04（S0句，全域）
  R-12（registry归属，高影响）
  R-15（glossary M01~M07，全站标准）
  R-16（cleanup 压缩，风险高）
  ≥5 页联动
  V7 基线/第二链路/RUN-next 相关
```

## 九、夜间与白天的分工建议

| 时间段 | 适合做的改动 | 适合做的分析 |
|--------|------------|------------|
| 白天（08:00~18:00） | 所有配方（R-01~R-16 全可）| 全部分析 |
| 晚间（18:00~00:00） | R-05~R-16（Controlled Zone 可）| 全部分析 |
| 深夜（00:00~06:00）| R-06~R-10,R-13~R-14（Safe）| S0/S1 分析、cleanup 分析 |
