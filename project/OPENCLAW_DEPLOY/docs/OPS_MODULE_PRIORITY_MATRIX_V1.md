# OPS_MODULE_PRIORITY_MATRIX_V1.md — 全站模块维护优先级矩阵

> 文档版本：v1.0（P3-42）
> 创建时间：2026-04-21
> 对应阶段：Phase 3 — P3-42 全站模块级 inventory、依赖关系、重复治理与模块维护底图功能包第一期
> 目的：建立模块维护优先级划分，让维护决策有据可依

---

## 一、优先级定义

| 优先级 | 定义 | 维护要求 | 典型场景 |
|--------|------|---------|---------|
| **P0** | 核心模块，不可乱动 | 改之前必须全站评估，改动后必须全站验证 | 真源层变更/结论性陈述/全局统一句 |
| **P1** | 高风险同步模块 | 改之前必须查依赖图，改动后必须同步相关模块 | 触发规则/判断矩阵/生命周期定义 |
| **P2** | 可局部优化模块 | 可以在本页面内独立优化，不需要全站同步 | 页面内局部说明/格式优化/文案统一 |
| **P3** | 可压缩/可合并观察模块 | 保持现状，有整合机会时再处理 | 近重复模块/冗长说明/可替代内容 |

**P0 模块的共同特征：**
- 是全局统一的"标准句"或"核心判断依据"
- 改动影响范围超过1个页面
- 是用户第一印象的来源
- 是系统级判断逻辑的锚点

## 六、优先级决策树

```
问自己：这个模块改动后会影响多少页面？
  ↓
影响 ≥2 个页面？
  → 是：再问：是全局标准句/核心判断/结论性陈述吗？
      → 是：→ P0（必须全站评估）
      → 否：→ P1（必须同步相关页面）
  → 否：再问：是局部说明/格式/训练提示吗？
      → 是：→ P2（可本页面独立优化）
      → 否：→ 问：这个内容是否已经高度重复在其他地方？
          → 是：→ P3（可压缩观察）
          → 否：→ P2
```

## 七、维护优先级检查工具

**每次修改模块前，先查这个清单：**

| 问题 | 答案 | 优先级 |
|------|------|--------|
| 这个模块在全站有统一标准句吗？ | 是 | P0 |
| 这个模块的结论会被其他模块引用为依据吗？ | 是 | P0 |
| 这个模块内容在多个页面有镜像副本吗？ | 是 | P0 或 P1 |
| 改这个模块需要查 OPS_CHANGE_IMPACT 吗？ | 是 | P1 |
| 这个模块属于"训练提示/页面说明/格式优化"吗？ | 是 | P2 |
| 这个模块内容高度重复在其他地方吗？ | 是 | P3 |
| 以上都不符合 | — | P2（默认） |
