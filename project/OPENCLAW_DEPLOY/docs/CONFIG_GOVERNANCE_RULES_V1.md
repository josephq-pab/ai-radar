## 一、配置来源优先级

配置值的生效优先级（从上到下，高优先级覆盖低优先级）：

| 优先级 | 配置类型 | 示例 | 生效说明 |
|-------|---------|------|---------|
| 1 | 代码硬编码 | top-k=5（build_analyst_opinions.py）| 无其他配置时使用 |
| 2 | 配置文件 | analyst_sources.json | 可覆盖代码默认值 |
| 3 | 脚本参数 | --top-k 3 | 运行时刻指定，覆盖配置文件 |
| 4 | 人工指定 | 用户临时指定运行参数 | 最高优先级，仅本次生效 |

**当前问题**：top-k/min-rel/B gate 阈值目前是代码硬编码，优先级1，没有配置入口。

**第一期目标**：将这些硬编码值明文化到配置状态页，但不改变其代码硬编码的性质。

## 二、配置生效规则

### 2.1 来源配置生效规则

| 配置变更 | 生效时机 | 验证方式 |
|---------|---------|---------|
| 修改 active=true→false | 下次运行时生效 | run-analyst-fetch.sh 验证 |
| 修改 seedUrls | 下次运行时生效 | dry-run 验证 |
| 新增来源 | 下次运行时生效 | dry-run + change-summary 验证 |

**生效原则**：来源配置变更后，必须执行 dry-run 验证，不允许只改配置不验证。

### 2.2 运行参数生效规则

| 配置变更 | 生效时机 | 验证方式 |
|---------|---------|---------|
| top-k 值修改 | 下次运行时生效（代码硬编码）| 记录在配置状态页，变更需改代码 |
| min-rel 值修改 | 下次运行时生效（代码硬编码）| 同上 |
| dry-run 模式 | 仅本次运行生效 | 脚本参数控制 |

**生效原则**：top-k/min-rel 当前是代码硬编码，第一期的"变更"是指在配置状态页中记录当前值，而不是提供在线编辑。

### 2.3 规则配置生效规则

| 配置变更 | 生效时机 | 验证方式 |
|---------|---------|---------|
| B gate 阈值修改 | 下次 rebuild_go_live_gate.py 时生效 | smoke_test.py 验证 |
| 双门槛/三档条件修改 | 下次 build 时生效 | rebuild_go_live_gate.py |

## 三、配置变更留痕要求

### 3.1 必须记录的配置变更

以下配置变更必须记录到 `data/config-change-log.md`：

| # | 配置项 | 变更类型 |
|---|------|---------|
| L1 | analyst_sources.json 的 active 状态变更 | 任何来源的启用/停用 |
| L2 | analyst_sources.json 的 seedUrls 变更 | 新增/删除 URL |
| L3 | top-k 值的变更 | 代码修改时 |
| L4 | B gate 阈值的变更 | 代码修改时 |
| L5 | 当前试用对象的变更 | 发送对象增减 |
| L6 | 当前内容基线版本的变更 | 冻结新版本时 |

### 3.2 配置变更记录格式

```markdown
## [日期] 配置变更记录

### 变更1
- 时间：[时间]
- 变更人：[变更人]
- 变更配置：[配置名称]
- 变更文件：[文件路径]
- 变更前：[变更前的值]
- 变更后：[变更后的值]
- 验证结果：[✅ 通过 / ❌ 失败]
- 备注：[原因/影响]
```

### 3.3 变更记录管理原则

- **先记录，再验证**：变更前先写入 config-change-log.md，验证通过才算正式生效
- **失败不掩盖**：验证失败时，在 config-change-log.md 中明确写"失败，原因：XXX"
- **不允许不记录的变更**：analyst_sources.json 的 active 状态变更必须有记录

## 四、试用期内配置修改边界

### 4.1 允许在试用期修改的配置

| 配置项 | 修改条件 | 验证要求 |
|--------|---------|---------|
| 来源 active 启停 | 任何时候 | dry-run 验证 |
| 试用对象 | 任何时候 | 通知用户 |
| 反馈问题措辞 | 非结构性变更 | 无需验证 |
| 措辞微调 | 不改变语义 | 无需验证 |

### 4.2 试用期禁止直接修改的配置

| 配置项 | 禁止原因 | 变更方式 |
|--------|---------|---------|
| top-k 硬编码值 | 需评估对输出的影响 | 必须通过 review 后决策 |
| B gate 阈值 | 影响 Gate 判断标准 | 必须通过 review 后决策 |
| confirmLevel 映射 | 影响内容分级标准 | 必须通过 review 后决策 |
| 双门槛数值（5000万/1000万）| 影响业务判断标准 | 必须通过 review 后决策 |

### 4.3 解冻条件（可修改禁止项）

以下条件满足时，允许修改"禁止直接修改"的配置：

- 有明确的负向反馈（≥2人反映该配置不合理）
- 经过 P3 系统化建设阶段的 review 决策
- 用户明确授权修改

## 五、配置变更审批机制

### 5.1 当前机制（轻量）

当前阶段采用**人工记录 + 验证通过生效**的轻量机制：

```
变更申请 → 记录到 config-change-log.md → 执行验证 → 确认生效/失败
```

### 5.2 审批权限

| 配置类型 | 当前审批人 | 说明 |
|---------|----------|------|
| 来源启停 | 用户 | 当前无 formal 审批流程 |
| 运行参数变更 | 用户 | 通过对话确认 |
| 规则阈值变更 | 用户 | 通过对话确认 |

### 5.3 未来扩展方向

当变更频率增加或多人协作时，可升级为：

- 变更申请 → Review → 审批 → 执行 → 验证 → 记录

但当前 MVP 阶段不需要。

## 六、第一期配置入口使用规范

### 配置状态页使用者行为规范

1. **查看配置状态**：直接打开 config-status.html
2. **发现配置值异常**：记录到 config-change-log.md，等待用户决策
3. **需要变更配置**：联系 agent 或用户，通过对话确认后修改
4. **不允许**：直接在 config-status.html 中编辑配置

### 配置状态页更新规范

- 每次配置变更后，更新 config-status.html 中的快照值
- 更新 config-change-log.md 中的变更记录
- 配置状态页更新不算"配置变更"，是快照刷新

## 七、配置状态页与 config-change-log.md 关系

|------|------|---------|
| config-status.html | 配置快照 | 配置变更后手动刷新 |
| config-change-log.md | 变更记录 | 每次配置变更时记录 |
