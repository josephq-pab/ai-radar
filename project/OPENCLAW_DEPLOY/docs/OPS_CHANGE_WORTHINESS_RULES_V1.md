## 一、变更价值判断总原则

**核心判断：收益 > 风险？**

每次变更请求，先问：
1. 这个改动解决了什么问题？
2. 这个问题不解决会怎样？
3. 改动引入的新风险有多大？
4. 收益和风险相比，值不值得？

**不修改也是一种方案。**

## 三、值得改的场景

以下改动**值得做**（有明确问题，不改影响使用，改了收益明确）：

| 改动场景 | 价值 |
|---------|------|
| 页面有错别字/病句，影响理解 | 改后用户不会困惑 |
| 模块标题与内容不符，用户找不到 | 改标题后用户能定位 |
| 路径推荐已过时，用户按路径走不通 | 改路径后用户能正常使用 |
| 触发条件描述错误，用户误解触发逻辑 | 改后用户不会误触发 |
| 配置页描述与实际配置不符 | 改后用户不会误解配置 |

## 四、收益风险评估标准

### 4.1 收益评估

| 收益维度 | 高收益 | 低收益 |
|---------|-------|-------|
| 用户理解 | 改后明显减少误判/困惑 | 改后无明显改善 |
| 系统稳定性 | 改后减少漂移风险 | 改后无显著改善 |
| 维护效率 | 改后减少维护复杂度 | 改后无显著改善 |

### 4.2 风险评估

| 风险维度 | 高风险 | 低风险 |
|---------|-------|-------|
| 影响页面数 | ≥3 页 | 1 页 |
| 涉及 S0/S1 句 | 是 | 否 |
| 涉及 P0 模块 | 是 | 否 |
| 涉及真源层 | 是 | 否 |
| 深夜改动 | 是 | 否 |

### 4.3 价值判断矩阵

| 收益/风险 | 低风险（1-2页/非S0/P0） | 高风险（≥3页/涉及S0/P0） |
|-----------|----------------------|------------------------|
| **高收益** | ✅ 值得改（Controlled） | ⚠️ 谨慎改（No-touch，建议白天） |
| **低收益** | ❌ 不值得（维持现状） | ❌ 不值得（维持现状） |

## 五、夜间任务边界

### 5.1 夜间适合做的事

| 适合夜间做的事 | 说明 |
|---------------|------|
| Safe Zone 改动（QC 即可完成） | 风险低，回归简单 |
| 文档更新（不涉及页面展示） | 不影响用户可见内容 |
| 版本标注更新 | 纯元信息更新 |
| 低风险样式/格式调整 | 不影响语义 |
| 治理文档新建/更新 | 不影响展示层 |

### 5.2 夜间不适合做的事

| 夜间不适合做的事 | 原因 |
|----------------|------|
| No-touch Zone 改动 | 高风险，需要完整 FC 和白天验收 |
| S0 句子改动 | 高风险，影响全局 |
| P0 模块改动 | 高风险，影响 ≥3 页 |
| 改动 ≥5 页的大范围变更 | 影响面大，夜间容易遗漏 |
| 真源层更新 | 真源层更新必须有展示层同步，夜间容易遗漏 |

### 5.3 夜间任务自检表

```
夜间准备执行变更前：

[ ] 改动属于 Safe Zone 吗？ → 是：继续 / 否：建议白天做
[ ] 改动涉及 S0/S1/P0 吗？ → 是：建议白天做
[ ] 改动影响页面数 ≥3 吗？ → 是：建议白天做
[ ] 当前时间是深夜（00:00~06:00）吗？ → 是：建议白天做
[ ] 以上全部为"否"：可以继续，但完成 QC 步骤
[ ] 以上任一为"是"：建议白天做，或完成 FC 步骤后继续
```

## 六、停止规则与升级条件

### 6.1 直接停止不继续改的情况

| 停止条件 | 说明 |
|---------|------|
| 改动请求不在本轮任务范围内 | 记录到 backlog，等下一轮 |
| 改动涉及 V7 基线/第二链路/RUN-next | 属于边界红线，当前阶段不改 |
| 改动会让系统变得更复杂 | 简单改复杂，禁止 |
| 没有明确改动理由 | 不知道改了解决什么问题，禁止 |
| 改动收益不明显 | 改了没大用，禁止 |
| 改动会破坏已有稳定功能 | 破坏稳定，禁止 |

### 6.2 升级到白天确认的情况

| 升级条件 | 说明 |
|---------|------|
| No-touch Zone 改动请求（确需执行） | 建议白天做，需要更多人工验收 |
| 改动影响 ≥5 页 | 影响面大，建议白天做 |
| S0 句改动（确需执行） | 建议白天做，需要完整 FC |
| 真源层更新（确需执行） | 必须白天，需要展示层同步 |

### 6.3 记入 cleanup backlog 的情况

| 记入 backlog 条件 | 处理方式 |
|-----------------|---------|
| 改动收益低，但确实有点问题 | 记入 OPS_REMAINDER_CLEANUP_PLAN_V1.md，等待后续处理 |
| 改动属于 Q-01~Q-10 类问题 | 已在 Q 表中标注，维持现状 |
| 改动需要更大范围重构 | 记入 backlog，不在当前阶段做 |

## 七、变更决策树

```
遇到变更请求：

Step 1：这个改动解决什么问题？
  → 说不清楚：停止，不需要改

Step 2：这个问题不解决会怎样？
  → 没什么影响：停止，维持现状

Step 3：这个改动涉及 S0/S1/P0 吗？
  → 是：建议白天做 / 继续判断是否值得
  → 否：继续

Step 4：改动影响多少页？
  → ≥5 页：建议白天做
  → 1~4 页：可夜间做，但完成 FC

Step 5：深夜吗？
  → 是，且涉及 No-touch/S0/P0/≥3页：建议白天做
  → 否：可以继续

Step 6：收益 > 风险？
  → 是：按对应 Zone 执行变更
  → 否：停止，记入 backlog
```
