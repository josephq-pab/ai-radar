## 一、高风险误用场景清单

以下场景是实际使用中最容易发生的误用。每个误用都标明了：错误做法 / 正确做法 / 为什么这是误用。

### 场景1：想知道当前要不要动，却先看证据页

- **错误做法**：直接打开 ops-evidence.html 找"有没有证据支持行动"
- **正确做法**：先看 single-chain-ops.html 判断"有没有新信号"，没有新信号就停在等待态
- **为什么是误用**：ops-evidence 是解释/补材料用的，不是判断"要不要动"的入口；先翻证据会跳过最重要的"有没有动作"判断
- **误用概率**：高
- **影响**：把判断路径倒过来，把辅助当成主路径

### 场景2：收到一句新消息，直接去决策页问"这个要不要处理"

- **错误做法**：看到任何新输入 → ops-decision.html → "帮我判断"
- **正确做法**：新输入 → ops-brief.html 快读判断（留痕/轻响应/评估？）→ single-chain-ops.html（如需要）→ 再决定是否去 ops-decision
- **为什么是误用**：ops-decision 是"边界不清/触发待判/异常待判"时才需要，不是所有新输入都要进决策；过度决策化会增加摩擦
- **误用概率**：高
- **影响**：把决策当成第一反应，而不是最后手段

### 场景3：想理解系统当前状态，先翻配置页

- **错误做法**：打开 config-status.html 从头读配置参数
- **正确做法**：先看 radar-home.html 状态横幅（一句话说清楚当前状态）
- **为什么是误用**：config-status 是参数/Gate 核对页，不是理解全局的入口；配置页会让使用者迷失在参数里，而抓不到业务重点
- **误用概率**：中
- **影响**：新接手者一上来就被参数淹没，觉得系统很复杂

### 场景4：把 ops-playbook 当成"想动手之前先想想"的思考页

- **错误做法**：还没进准备阶段就开始翻 playbook，"看看要做什么"
- **正确做法**：先在 single-chain-ops.html 判断清楚当前状态，确认要执行/准备中，才去 ops-playbook 查执行步骤
- **为什么是误用**：playbook 是执行步骤指引，不是思考框架；在还没确认要执行时翻 playbook 会导致"想了半天但没有结论"或"以为已经开始准备了"
- **误用概率**：高
- **影响**：把"想动手"当成"已经在准备"，误判系统状态

### 场景5：看到"同意准备"就以为 RUN-next 已经启动

- **错误做法**：看到升级回复中"同意准备" → 直接翻 ops-playbook 查执行步骤
- **正确做法**：同意准备 = 进入准备态(P)，RUN-next 仍需人工确认（V7 基线冻结）
- **为什么是误用**：准备态(P) ≠ RUN-next 已触发；两者之间还有人工确认步骤；把"同意准备"当成 RUN-next 已开启会误判为"已经在跑"
- **误用概率**：高
- **影响**：误判 RUN 状态，可能跳过关键人工确认步骤

### 场景6：看到"补材料"就以为申请被否了

- **错误做法**：看到升级回复"补材料" → 当成否定，不继续跟进了
- **正确做法**：补材料 = 信息不够，不是否决；补充完整后可继续进入评估/准备
- **为什么是误用**：把"信息不够"当成"结论否定"，会错误放弃本来可以推进的输入
- **误用概率**：高
- **影响**：有效输入被错误放弃，系统失去响应能力

### 场景7：看到候选来源就以为第二链路可以动了

- **错误做法**：发现新的分析师/货币/债券来源候选 → 直接翻 ops-evidence 找证据，准备启用
- **正确做法**：来源候选必须经过评估 → 人工确认 → 才能成为可用来源；第二链路当前冻结
- **为什么是误用**：候选 ≠ 可用；把候选当成解冻，会违反口径一致性；第二链路解冻需要结构性来源，不只是候选
- **误用概率**：中
- **影响**：违反口径一致性，可能产生不合规数据

### 场景8：把 ops-registry 当首页，直接在 registry 里找业务结论

- **错误做法**：打开 ops-registry.html 期望找到"当前状态是什么"的答案
- **正确做法**：ops-registry 是真源/版本/元信息入口，业务状态判断在 radar-home / ops-brief / single-chain-ops
- **为什么是误用**：registry 管的是"各版本的依据是谁"，不是"现在要不要动"；把 registry 当首页会找不到业务结论
- **误用概率**：中
- **影响**：使用者以为系统缺少业务信息，实际只是进错页面

### 场景9：把 ops-routes 当规则原文背诵

- **错误做法**：把 ops-routes.html 打印出来背诵"按角色应该先看什么"
- **正确做法**：ops-routes 是路径选择工具，遇到场景时来查"我现在该走哪条路"，不需要提前背诵
- **为什么是误用**：routes 是索引，不是内容；背诵 routes 而不理解各页角色，遇到新场景还是不知道怎么处理
- **误用概率**：低
- **影响**：学会路径但不会判断场景

### 场景10：把 ops-glossary 当第一本教材从头读

- **错误做法**：新接手第一件事是打开 ops-glossary.html 从头学到尾
- **正确做法**：ops-glossary 是"遇到术语再查"的工具，不是学习路径；先走通最小训练路径（radar-home → ops-routes → ops-brief），再遇到术语时查 glossary
- **为什么是误用**：glossary 是参考工具，不是教材；从头读 glossary 会抓不到重点，且容易迷失在术语里
- **误用概率**：中
- **影响**：新使用者觉得系统太复杂，实际只是学习方法不对

### 场景11：收到"暂不推进"就直接放弃

- **错误做法**：看到升级回复"暂不推进" → 当成否定，不再跟进
- **正确做法**：暂不推进 = 维持等待态(W)，不是关闭可能性；条件变化后可再提
- **为什么是误用**：暂不推进是"当前不推进"，不是"以后也不能推进"；当成否定会错误放弃有效的输入
- **误用概率**：高
- **影响**：有效输入被错误放弃

### 场景12：快报里看到新信号就以为要立即升级

- **错误做法**：ops-brief 里有新信号 → 立即去 ops-decision 要求升级
- **正确做法**：新信号 → 先在 ops-brief 判断类型（留痕/轻响应/评估？）→ 再决定是否需要 ops-decision
- **为什么是误用**：大部分新信号只需要留痕(L)或等待(W)，不需要升级(E)；过度升级会浪费决策资源
- **误用概率**：中
- **影响**：把轻微信号当成紧急信号，消耗决策资源

## 二、哪些页面不是起点（按使用时机）

### ops-evidence.html（证据页）
- ❌ 不是起点：第一次上手不要先翻证据页
- ✅ 正确时机：需要解释 / 补材料 / 证明"为什么判断成立"时才来

### ops-decision.html（决策页）
- ❌ 不是默认入口：不是每轮都要来
- ✅ 正确时机：边界不清 / 触发待判 / 异常待判时才来

### ops-playbook.html（执行流程页）
- ❌ 不是思考页：还没进准备/执行阶段不要先翻
- ✅ 正确时机：已确认要执行 / 已在准备中 / 已在执行中

### config-status.html（配置页）
- ❌ 不是理解全局的起点：不要一上来就读配置
- ✅ 正确时机：核参数 / 对比配置 / 检查 Gate

### ops-glossary.html（术语页）
- ❌ 不是教材：不要从头读
- ✅ 正确时机：遇到术语时查疑用

### ops-registry.html（真源登记页）
- ❌ 不是首页：不要当成业务状态入口
- ✅ 正确时机：冲突时查"依据是谁" / 版本核对

## 三、哪些页面不是每次必看

| 页面 | 不是每次必看的理由 |
|------|-----------------|
| ops-decision.html | 只有边界不清才需要，不需要每轮都进 |
| ops-playbook.html | 只有进入执行/准备阶段才需要 |
| ops-evidence.html | 只有需要解释/补材料时才需要 |
| config-status.html | 只有核参数时才需要 |
| ops-registry.html | 只有查真源/版本时才需要 |
| ops-glossary.html | 只有查术语时才需要 |

## 四、正确替代路径对照

| 误用冲动 | 正确替代 |
|---------|---------|
| 想知道要不要动，却去翻证据页 | 先看 single-chain-ops 的"现在要不要动" |
| 新输入直接去决策页 | 先去 ops-brief 快读判断 |
| 想理解系统状态，先翻配置页 | 先看 radar-home 状态横幅 |
| 还没准备就想看执行步骤 | 先确认在 single-chain-ops 的当前停点 |
| 看到同意准备以为 RUN-next 已开 | 记住：准备态 P ≠ RUN-next，RUN-next 仍需确认 |
| 看到补材料以为被否了 | 记住：补材料=信息不够，补完可以继续 |
| 看到候选来源就想解冻第二链路 | 记住：候选≠可用，第二链路仍冻结 |
| 把 registry 当首页找结论 | 记住：业务结论在 ops-brief/single-chain-ops |
| glossary 从头读 | 记住：glossary 是查疑工具，不是学习路径 |
| 暂不推进直接放弃 | 记住：暂不推进=维持等待，条件变可再提 |
