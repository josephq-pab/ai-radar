> 创建时间：2026-04-21

> 目的：建立"改了什么内容之后该查谁、怎么算没改坏"的标准回归剧本

---

## 一、回归剧本使用说明

**使用场景：**
- 每次有变更请求时，先判断改动类型和风险等级
- 按本剧本的对应章节执行回归检查
- 回归完成后填写验收清单

**两套回归版本：**
- **快速回归版（Quick Check）**：适合 Safe Zone / Controlled Zone 的小型改动
- **完整回归版（Full Check）**：适合 No-touch Zone / S0 句 / P0 模块改动

**回归原则：**
1. 先确认值不值得改，再决定回归深度
2. 先查依赖图，再决定回归范围
3. 先改真源，再改镜像
4. 高风险改动不改夜间随意执行

## 二、快速回归剧本（Quick Check）

### 2.1 适用场景

- 🟢 Safe Zone 改动（模块标题/路径引导/样式类）
- 🟡 Controlled Zone 中低风险改动（S2/S3 句/局部段落）
- 改后"看起来正常"即可的简单调整

### 2.2 快速回归步骤

**Step QC-1：改后自查（本页面）**
```
[ ] 打开改动后的页面
[ ] 确认页面可正常加载
[ ] 确认改动位置上下文中无明显异常（颜色/字体/布局未破坏）
[ ] 确认改动句子上下文通顺
```

**Step QC-2：最近邻同步检查（相关页面 ≤2 个）**
```
[ ] 确认相关页面的对应引用处是否一致
[ ] 如有 banner/footer 引用，确认是否需要同步
[ ] 检查各页面版本标注（P3-xx）是否需要更新
```

**Step QC-3：判断是否需要升级到完整回归**
```
[ ] 改动涉及 S0/S1 句吗？ → 是：升级到完整回归
[ ] 改动影响 ≥3 页吗？ → 是：升级到完整回归
[ ] 改动涉及 P0 模块吗？ → 是：升级到完整回归
[ ] 以上皆否：快速回归完成 ✓
```

### 2.3 快速回归验收清单

| # | 检查项 | 结果 |
|---|--------|------|
| QC-1 | 页面可正常加载 | ✅/❌ |
| QC-2 | 改动句子上下文通顺 | ✅/❌ |
| QC-3 | 相关页面引用已同步 | ✅/❌ |
| QC-4 | 版本标注已更新（如需） | ✅/❌ |
| QC-5 | 无引入新的明显错误 | ✅/❌ |

## 三、完整回归剧本（Full Check）

### 3.1 适用场景

- 🔴 No-touch Zone 改动
- S0 句子改动
- P0 模块改动
- 涉及 ≥3 页面的 S1 句改动
- 生命周期/第二链路/V7 冻结相关改动

### 3.2 完整回归步骤

**Step FC-1：改动类型确认**
```
[ ] 确认改动属于哪个 Type（Type-A S0句 / Type-B S1句 / Type-D 模块标题 / Type-E 段落）
[ ] 确认风险等级（No-touch / Controlled / Safe）
[ ] 确认涉及页面数量
```

**Step FC-2：真源层优先检查**
```
[ ] 确认真源文档（STAGE2/OPS_COPY_BASELINE/OPS_LIFECYCLE_MAP）是否需要同步更新
[ ] 如真源层也有对应内容，确认是否已同步更新
[ ] 真源未同步，禁止推进展示层回归
```

**Step FC-3：依赖图检查（必做）**
```
[ ] 打开 OPS_MODULE_DEPENDENCY_MAP_V1.md
[ ] 找到改动对应的模块
[ ] 列出"高依赖模块"和"跨页热点"
[ ] 确认改动影响的所有下游展示模块
```

**Step FC-4：全站镜像同步检查**
```
[ ] 根据 OPS_MODULE_DEPENDENCY_MAP，确认哪些页面有该模块的镜像副本
[ ] 逐页确认镜像内容是否已同步
[ ] 特别检查 banner/footer（最容易遗漏）
```

**Step FC-5：S0/S1 句专项检查（仅 S0/S1 改动时）**
```
[ ] 打开 OPS_SENTENCE_REGISTRY_V1.md
[ ] 确认该句子在其他页面的所有出现位置
[ ] 确认 OPS_SENTENCE_DIFF_MATRIX 中该句的差异状态
[ ] 确认"允许差异"和"必须同步"的边界
```

**Step FC-6：P3 版本标注更新**
```
[ ] 确认所有被改动的页面版本标注已更新
[ ] 确认 docs/OPS_MODULE_INVENTORY_V1.md 中对应模块的版本已更新
[ ] 确认 docs/OPS_SENTENCE_REGISTRY_V1.md（如涉及句子改动）
```

**Step FC-7：回归后人工验收**
```
[ ] 在本地/测试环境打开所有涉及页面
[ ] 逐页检查：改动句子在页面语境中是否通顺/准确
[ ] 检查页面整体布局是否未因改动被破坏
[ ] 确认 banner/footer 内容与页面主体内容一致
```

### 3.3 完整回归验收清单

| # | 检查项 | 结果 |
|---|--------|------|
| FC-1 | 改动类型和风险等级已确认 | ✅/❌ |
| FC-2 | 真源层已同步（如涉及） | ✅/❌ |
| FC-3 | OPS_MODULE_DEPENDENCY_MAP 已查 | ✅/❌ |
| FC-4 | 全站镜像副本已同步 | ✅/❌ |
| FC-5 | S0/S1 句其他位置已确认 | ✅/❌ |
| FC-6 | P3 版本标注已更新 | ✅/❌ |
| FC-7 | 人工验收通过 | ✅/❌ |

## 四、风险等级对应的回归深度

| 风险等级 | 回归版本 | 回归步骤数 | 最低要求 |
|---------|---------|-----------|---------|
| 🟢 Safe Zone | Quick Check | 3步+5项检查 | QC-1 + QC-2 |
| 🟡 Controlled Zone | Quick Check → Full Check（按需升级） | 3步+5项 或 7步+7项 | QC-1 + QC-2 + QC-3 |
| 🔴 No-touch Zone | Full Check（必须） | 7步+7项 | FC-1 ~ FC-7 全做 |

## 五、不同改动类型的回归重点

### 5.1 S0 句子改动（Type-A）

**回归重点：** 全站镜像同步 + S0 句一致性
```
必须做：
  1. 查 OPS_SENTENCE_REGISTRY_V1.md，确认 S0 句所有出现位置
  2. 查 OPS_SENTENCE_DIFF_MATRIX，确认当前各页差异状态
  3. 同步所有镜像副本（banner/footer/各页模块）
  4. 确认改后句子在每页语境中仍然通顺
  5. 填写 FC-1 ~ FC-7 全部验收项
禁止：
  1. 夜间单独改一个页面，不同步其他页面
  2. 只改 banner 不改主体内容
  3. 改完不做人工验收
```

### 5.2 S1 句子改动（Type-B）

**回归重点：** 依赖图 + 相关页面同步
```
必须做：
  1. 查 OPS_MODULE_DEPENDENCY_MAP，确认高依赖模块
  2. 确认涉及页面数量（≥3则升级到完整回归）
  3. 同步所有相关页面
禁止：
  1. 只改一个页面，不检查其他引用处
```

### 5.3 模块标题改动（Type-D）

**回归重点：** 引用链检查
```
必须做：
  1. grep 全站搜索该标题，确认无其他引用
  2. 如有引用，确认引用处是否需要同步
```

### 5.4 段落结构/顺序改动（Type-E）

**回归重点：** 上下文完整性
```
必须做：
  1. 确认改动不破坏段落的因果/逻辑关系
  2. 确认上下段落衔接自然
  3. 确认改动后段落总长度未异常增长/缩短
```

## 六、改动后验收问题清单

每次回归完成后，必须能回答以下问题：

### 基础问题（必须全部可答）
```
[ ] 改了什么？（具体句子/模块标题/段落位置）
[ ] 改了哪个页面？（原始页 + 涉及的所有镜像页）
[ ] 是否有真源层同步？（STAGE2/OPS_COPY_BASELINE）
[ ] S0/S1 句是否已全站同步？（逐页确认）
[ ] 版本标注是否已更新？
[ ] banner/footer 是否已同步？
```

### 高级问题（S0/P0/No-touch 改动必须回答）
```
[ ] 为什么这个改动值得做？（收益 > 风险）
[ ] 改动是否会影响用户的判断逻辑？
[ ] 改动是否会引发新的误判风险？
[ ] 是否有备选方案（不改也是一种方案）？
[ ] 夜间改动是否合适？（No-touch 建议白天做）
```

## 七、停止规则

**以下情况直接停止，不继续回归：**

| # | 停止条件 | 原因 |
|---|---------|------|
| 1 | 改动涉及 S0 句，但未完成全站同步检查 | S0 句未全站同步是高风险漂移 |
| 2 | 真源层与展示层不同步 | 真源未同步，展示层迟早会乱 |
| 3 | 改动后人工验收发现语义发生变化 | 语义变化意味着误判风险增加 |
| 4 | 改动影响 ≥5 页，但改动时间在深夜 | 深夜改动 + 大范围影响 = 高风险，建议白天做 |
| 5 | 改动涉及 V7 基线/第二链路/RUN-next 冻结相关内容 | 这些内容属于 No-touch Zone，建议白天人工确认 |
