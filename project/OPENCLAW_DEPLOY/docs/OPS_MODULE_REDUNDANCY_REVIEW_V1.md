## 一、重复分析前提说明

**本清单的重复判定标准：**
- **完全重复（Identical）：** 同一内容在多个模块中出现，且完全相同
- **近重复（Near-Duplicate）：** 同一意思在多个模块中表达，内容高度相似但表述略有差异
- **角色性重复（Intentional）：** 同一意思在不同页有不同角色定位，保留有明确价值
- **可压缩（Compressible）：** 内容有价值但表达方式过于冗长，可以精简

**本清单不包括：**
- 跨页镜像（状态横幅等有意的全站统一展示，不计为重复）
- 真源到展示层的派生关系（STAGE2 → 各页面，是正常的派生不是重复）

## 三、近重复模块

以下模块存在近重复（高度相似但表述有差异），需要审计是否需要统一：

### 3.1 【风险区】误判澄清类近重复

**近重复组A：**
- home-M24：相似场景边界对照（8组）
- brief-M19：高频误判对照（6组）
- brief-M16：不应误解事项（4项）
- ops-M20：相似场景边界判定（8组，ops视角）
- decision-M10：边界场景对照（8组）
- decision-M22：哪些边界对值得进决策页（6组）

**问题描述：** 
- "相似场景边界对照" / "高频误判对照" / "不应误解事项" / "边界场景对照" 本质上都是"防止误判"的内容
- 六处各自维护，内容高度重叠但表述角度不同

**分析结论：**
- 当前状态：可接受（各自角度略有不同，都是防误用的内容）
- 风险等级：中（维护负担高，容易出现漂移）
- 建议：未来将六处整合为全站统一的"高频误判澄清库"，各页做引用而非独立维护
- 本轮不压缩（风险太高，需要重新训练用户心智）

**本轮动作：** 标注为"近重复-观察"，暂不压缩

## 四、必须保留的角色性重复

以下重复是刻意设计，压缩会伤害可用性：

| 模块组 | 重复理由 | 结论 |
|--------|---------|------|
| 状态横幅全站镜像 | 全站统一状态展示，保障一致性 | 保留，同步维护 |
| 首页能力看板 vs 快报能力看板 | 受众不同：首页面向首次访问，快报面向回访 | 保留 |
| 首页生命周期 vs ops状态流转 | 维度不同：全局概览 vs 运营视角 | 保留 |
| 首页典型场景 vs ops判例 vs 路径场景 | 同一场景三页各有侧重（总览/运营/路径） | 保留 |
| 各页元信息块 | 各页独立治理说明，页面级个性化 | 保留 |

## 五、可压缩模块候选

以下模块满足"内容有价值但表达冗长，可以精简"的条件：

### 5.1 ops-registry.html（真源登记页）— 高度重复映射

**问题：** registry-M14~M17（值班结果/升级/拍板/阶段与真源映射）四节高度重复
- 都是"某种运营对象 → 归属真源文档"的结构
- 内容逻辑相同，仅对象不同

**可压缩方案：**
- 将四节合并为"运营对象真源归属总表"
- 用统一表格替代四节重复结构
- **本轮状态：** 低优先级，不在本轮实施

### 5.2 home-M12（自上次以来变化）— 信息时效性强但维护价值有限

**问题：** 这个模块的"delta"信息时效性很短，每次更新都要改
- 但它的功能（告诉用户什么变了）在快报页（brief-M13回访用户）已有更好表达

**可压缩方案：**
- 将 home-M12 精简为"指向 brief-M13 的引用说明"
- **本轮状态：** 低优先级，不在本轮实施

### 5.3 各页 P3-40 训练提示区块

**问题：** 每个页面底部都有一个 P3-40 训练提示区块，格式相似但内容各自独立
- 这些区块本质上是重复的（都是"如何正确使用本页面"的说明）

**可压缩方案：**
- 建立全站统一的"页面训练提示模板"
- 各页引用而非独立维护
- **本轮状态：** 低优先级，牵涉面广，不在本轮实施

## 七、重复治理建议原则

**原则1：角色性重复保留**
同一内容在不同页面因受众/场景不同而有不同表述，这是合理设计，不压缩。

**原则2：近重复监控**
同一意思在多处表达且表述有差异时，以一处为准，其他处做引用说明。

**原则3：禁止维护性重复**
不应出现"同一内容在两处独立维护且没有任何差异化价值"的情况。如发现，优先合并。

**原则4：跨页镜像必须同步**
状态横幅等刻意设计的跨页镜像，维护时必须全站同步，不得遗漏。

**原则5：本轮不引入大规模重构**
本轮重复治理的目标是识别和记录，不是压缩。压缩在有充分用户测试的情况下才能推进。
