## 背景

P3-34 已完成升级说明与人工拍板请求规则，运营者现在知道"何时升级、怎么写升级说明"。

但仍存在一个问题：

> **拍板人给出回复后，运营者如何最小吸收、分类、回写并决定下一步停点？**

当前摩擦：
- "先观察"被误当成"同意推进"
- "补材料"被误当成"否决"
- "不同意本轮推进"被误当成"永久否决"
- 回复后不知道该停在哪
- 为了回写一个回复，又重新看很多页

本功能包定义拍板回复分类、回写模板、回复后停点规则。

---

## 拍板回复分类定义

### A. 同意推进

**适用**：明确同意进入下一步评估/准备

**回复识别信号**：
- "可以" / "同意" / "推进" / "可以继续" / "按建议执行"
- "同意进入评估" / "同意进入准备" / "可以开始准备"

**默认动作**：
1. 回写"已获同意" + 对应时间
2. 进入升级说明中描述的下一步（评估→继续评估，或准备→进入准备检查）
3. **不等于触发 RUN-next**：只是获得推进许可，不代表 RUN-next 已执行

**停点**：进入下一步评估或准备检查，不自动触发 RUN-next

---

### B. 暂不推进 / 维持当前

**适用**：明确表示本轮先不动、继续等待、维持观察

**回复识别信号**：
- "先观察" / "继续关注" / "维持现状" / "暂不推进" / "本周先不动"
- "等下次信号明确再说" / "暂时搁置"

**默认动作**：
1. 回写"本轮维持等待态 / 暂不推进 / 继续观察"
2. 停止，不继续深查，不继续写材料

**停点**：回到等待态，下次值班时重新进入日常巡检路径

**重要区分**：
- 暂不推进 ≠ 永久否决
- 下次出现明确信号时可重新发起升级
- 不需要每次值班都报告"还没推进"

---

### C. 需补材料 / 需补说明

**适用**：不是否决，而是要求补充理由/依据/范围

**回复识别信号**：
- "依据不够" / "补一下理由" / "说明一下范围" / "依据不足"
- "请补充" / "需要再看" / "再提供一下"
- "条件不够充分" / "还需要说明"

**默认动作**：
1. 回写"需补材料 + 具体补什么"
2. 回到评估状态，补充最小必要依据（1~2个）
3. 补充后重新发起升级（不需要重新写完整升级说明）

**停点**：回到升级说明的评估状态，补充后重新升级，不进入准备

**重要区分**：
- 需补材料 ≠ 否决
- 补完材料后可以重新升级，不是重新走整套流程
- 不需要写长篇补充报告

---

### D. 不同意 / 否决本次建议

**适用**：明确否定当前建议或当前路径

**回复识别信号**：
- "不同意" / "不采纳" / "暂不考虑" / "否决"
- "不符合条件" / "不同意推进"

**默认动作**：
1. 回写"本次不采纳 + 原因（如有）"
2. 停止，不继续推进
3. 必要时归入开放项（OPS_OPEN_ITEMS_V1.md）

**停点**：直接停止，本次升级闭环

**重要区分**：
- 本次不同意 ≠ 永久否决
- 若下次有新的明确信号，可重新发起升级（注明"基于新信号重新升级"）
- 不需要每次值班都报告"上次不同意"
- 不自动重启流程，不自动解冻第二链路

## 回写模板

### 一句话模板（快速回写）

| 回复类型 | 一句话格式 |
|---------|----------|
| 同意推进 | `[日期] 已获同意。进入[评估/准备]。不等同于RUN-next已触发。` |
| 暂不推进 | `[日期] 本轮维持等待态。暂不推进。继续观察。` |
| 需补材料 | `[日期] 需补材料：[补什么]。回到评估状态。` |
| 不同意本次 | `[日期] 本次不采纳[原因如有]。本次升级闭环。下次有明确信号可重新升级。` |

### 2~3行模板（标准回写）

```
[日期] 拍板回复回写。

【收到回复】：[同意推进 / 暂不推进 / 需补材料 / 不同意本次]
【回写结论】：[对应结论]
【下一步】：[进入下一步 / 维持等待 / 补材料 / 停止]
【是否重新升级】：[是/否，取决于回复类型]
```

### 回复类型与停点速查

| 回复类型 | 停在哪 | 是否继续扩写 | 是否重新升级 |
|---------|-------|------------|------------|
| 同意推进 | 进入下一步评估或准备 | 不继续扩写 | 不需要 |
| 暂不推进 | 等待态 | 不继续扩写 | 下次有明确信号时重新升级 |
| 需补材料 | 评估状态（补充依据后） | 只补最小必要依据 | 补充后重新升级 |
| 不同意本次 | 停止 | 不继续扩写 | 下次有新信号时重新升级 |

## 收到回复后的最小动作路径

```
收到升级回复
    ↓
判断回复类型
    ↓
    ├─ 同意推进 → 回写"已获同意" → 进入下一步评估/准备
    ├─ 暂不推进 → 回写"维持等待" → 停止，回到等待态
    ├─ 需补材料 → 回写"需补X" → 补最小必要依据 → 重新升级
    └─ 不同意本次 → 回写"本次不采纳" → 停止，归入开放项（如必要）
    ↓
停在对应节点，不继续扩写材料
```

## 什么不属于结论性回复

| 回复 | 正确理解 | 错误理解 |
|------|---------|---------|
| "先观察" | 暂不推进，维持等待 | 等于同意推进 |
| "补一下理由" | 需补材料，回到评估 | 等于否决 |
| "不同意本次" | 本次不采纳，停止 | 等于永久否决 |
| "等下次信号" | 暂不推进，下次升级 | 需要每次报告进展 |
| "条件不够充分" | 需补材料，不是否决 | 等于不批准 |

---

## 当前不应误解的事项

1. **回复不等于自动执行**：同意推进只是获得许可，不代表 RUN-next 已触发或已执行。
2. **暂不推进不是永久否决**：下次有明确信号时可重新升级，不需要每次值班都报告。
3. **需补材料不是否决**：补完后可以重新升级，不是重新走整套流程。
4. **不同意本次不是永久否决**：若下次有新的明确信号，可重新发起升级，注明"基于新信号重新升级"。
5. **收到回复后不需要重新看所有页面**：先分类，再决定是否需要查证，依据 ops-evidence.html / ops-registry.html 补最小必要依据即可。
6. **大多数回复直接停止即可**：只有"需补材料"需要回到评估状态补充，其他类型直接停。

## 何时需要回到评估页（决策页）

| 情形 | 是否回到评估页 |
|------|-------------|
| 需补材料 / 需补说明 | ✅ 是（补充后重新升级）|
| 回复附带新条件或新范围 | ✅ 是（需要重新理解新条件）|
| 边界仍不清，需要再判断 | ✅ 是（回到 ops-decision.html）|
| 单纯"同意推进" | ❌ 否（直接进入下一步，不继续在决策页深查）|
| "暂不推进" | ❌ 否（停止，不继续在决策页深查）|
| "不同意本次" | ❌ 否（停止，不继续在决策页深查）|
