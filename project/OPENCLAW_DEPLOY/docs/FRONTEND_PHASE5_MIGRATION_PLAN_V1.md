## 一、迁移目标

从当前 Phase 4 静态治理基线（十页 HTML + 十六份治理文档），迁移到可维护的模板/组件体系，**不改变页面视觉和治理语义**。

### 1.1 迁移原则
1. **内容不变**：页面视觉、治理语法、层级结构保持不变
2. **结构抽象**：将共性 Block 归纳为模板和组件
3. **渐进实施**：分阶段执行，不搞大爆炸式重写
4. **可回滚**：每阶段有明确的回滚边界

### 阶段 2：PHASE5-2 — 模板映射标注

**目标**：在页面 HTML 中添加模板归属注释，不改变页面内容

**交付物**：
- 十页 HTML 添加 `<!-- TEMPLATE: T-MAINCHAIN -->` 等注释
- 每页的 Block 序列以注释形式标注
- 组件候选以 `<!-- COMPONENT-CANDIDATE: FirstScreenJudge -->` 形式标注

**执行顺序**：
1. 先标注 radar-home（T-HOME）
2. 再标注 single-chain-ops（T-MAINCHAIN）
3. 再标注 ops-decision（T-DECISION）
4. 再标注 ops-playbook（T-PLAYBOOK）
5. 最后标注支援页（T-SUPPORT）

**风险控制**：
- 只加注释，不改任何内容
- 每页标注后立即验证页面仍可正常打开

---

### 阶段 3：PHASE5-3 — 轻量共性 Block 抽象

**目标**：将四个最高优先级组件抽象为可复用片段（SSI/partial 或框架无关的模板片段）

**优先级**：
1. `FirstScreenJudge`（首屏判定区）
2. `FooterNav`（Footer 导航）
3. `GovernanceFold`（治理折叠区）
4. `PageMeta`（页面 Meta）

**执行顺序**：
1. 从支援页（ops-glossary/config-status）开始试点
2. 验证片段引用正常后，再推广到主链路页
3. 首页 radar-home 最后处理（风险最高）

**风险控制**：
- 每个页面提供原始 HTML 的备份（`*.html.bak`）
- 片段引用失败时，页面降级展示原始内容（不报错）

---

### 阶段 4：PHASE5-4 — 组件接口实施

**目标**：建立简单的组件接口机制（无框架依赖）

**组件接口形式**（推荐两种方案）：

**方案 A**：HTML Template + JS 渲染
```html
<template id="tpl-first-screen">
  <!-- 首屏判定区模板 -->
</template>
<script>
  function renderFirstScreen(container, questions) {
    // ...
  }
</script>
```

**方案 B**：纯 HTML SSI/Partial
```html
<!--#include virtual="/components/first-screen.html" -->
```

**推荐方案 A**（无服务器依赖，纯前端）

---

### 阶段 5：PHASE5-5 — 页面模板固化

**目标**：将模板归属固化为页面元数据的一部分

**交付物**：
- 每页添加 `data-template="T-MAINCHAIN"` 属性
- 页面 meta 区记录模板类型
- 建立页面-模板对照表

## 四、暂不处理项

以下事项在 Phase 5 期间**不处理**，留待 Phase 6：

| 项 | 原因 |
|----|------|
| 框架迁移（React/Vue）| 当前页面为静态 HTML，框架迁移成本高，Phase 5 不做 |
| 交互增强 | 页面治理目标是"可理解"，不是"可交互" |
| 数据动态化 | 静态快照，不做数据拉取能力 |
| 移动端适配 | 当前为桌面只读场景，移动端优先级低 |
| 国际化 | 当前为中文场景，暂不支持多语言 |

## 五、风险控制矩阵

| 阶段 | 主要风险 | 控制措施 | 回滚方式 |
|------|---------|---------|---------|
