# OPS_MODULE_MAINTENANCE_GUIDE_V1.md — 全站模块维护指南

> 文档版本：v1.0（P3-42）
> 创建时间：2026-04-21
> 对应阶段：Phase 3 — P3-42 全站模块级 inventory、依赖关系、重复治理与模块维护底图功能包第一期
> 目的：建立"改动某个模块时该查什么、怎么改、怎么验收"的标准维护步骤

---

## 一、本指南的适用对象

本指南适用于以下维护场景：
- 修改某个模块的内容（不改模块边界）
- 修改某模块的表述/句式
- 发现某模块与其他模块不一致
- 发现某模块内容已过期

本指南不适用于：
- 新增页面（P3-41 已建立强边界，不新增独立入口页）
- 修改生命周期逻辑（真源层改动，走专项流程）
- 脚本/analyst_sources.json 修改（真源层改动，走专项流程）

## 二、模块改动的标准顺序

### 第1步：识别模块并查优先级

```
先回答：我要改哪个模块？
  ↓
查 OPS_MODULE_INVENTORY_V1.md 确认模块 ID 和名称
  ↓
查 OPS_MODULE_PRIORITY_MATRIX_V1.md 确认优先级
```

### 第2步：识别真源和依赖

```
查 OPS_MODULE_DEPENDENCY_MAP_V1.md
  ↓
回答：这个模块的上游真源是什么？
  ↓
回答：改这个模块，哪些下游模块会受影响？
```

### 第3步：判断是否需要全站同步

```
P0 模块？
  → 改之前必须：全站评估 + STAGE2 确认 + 全站同步 + REVIEW_LOG

P1 模块？
  → 改之前必须：查依赖图 + 同步相关页面 + REVIEW_LOG

P2 模块？
  → 本页面内修改即可，仍需：REVIEW_LOG + CHANGE_CONTROL

P3 模块？
  → 观察优先，不一定需要修改
```

### 第4步：执行修改

按以下顺序执行：

```
1. 先改真源层（如涉及真源层改动）
2. 再改展示层（如涉及展示层改动）
3. 最后验证一致性
```

### 第5步：验证

```
对照 OPS_MODULE_DEPENDENCY_MAP_V1.md 验证依赖关系
  ↓
检查下游展示是否已同步
  ↓
运行最小验证清单（见第五章）
```

### 第6步：同步跟踪文档

```
必须更新：
  □ CHANGE_CONTROL.md（每次改动）
  □ REVIEW_LOG.md（每次改动）

可能需要更新：
  □ OPS_MODULE_INVENTORY_V1.md（如模块有变更）
  □ OPS_MODULE_DEPENDENCY_MAP_V1.md（如依赖关系有变更）
  □ OPS_MODULE_REDUNDANCY_REVIEW_V1.md（如重复关系有变更）
  □ FEEDBACK_TRACKING_STATUS.md（如涉及状态主线变化）
```

## 三、各优先级模块的维护要求

### 3.1 P0 模块维护要求

**改前必须完成：**

1. **全站影响评估**
   - 列出所有受影响页面（P0 模块影响范围通常 ≥3 页）
   - 列出所有受影响文档

2. **真源确认**
   - 确认改动是否符合 STAGE2 的当前状态
   - 如 STAGE2 也需要改，先改 STAGE2

3. **同步清单制定**
   - 制定完整的"先改哪个、再改哪个"顺序

4. **测试验证计划**
   - 制定改完后的验证步骤

**改后必须完成：**

1. 全站 P0 相关页面同步
2. OPS_MODULE_DEPENDENCY_MAP 更新（如依赖关系变化）
3. CHANGE_CONTROL.md + REVIEW_LOG.md 更新
4. 通知相关方（如果有人正在使用该模块作为依据）

**示例场景：** 修改 ops-M06（次轮触发规则摘要）
- 先查依赖：影响 brief-M15、decision-M05（触发条件引用）
- 改之前：确认 STAGE2 的触发规则内容
- 改之后：同步修改 brief-M15 和 decision-M05

---

### 3.2 P1 模块维护要求

**改前必须完成：**

1. **依赖图检查**
   - 列出直接相关的 P1/P2 页面
   - 确认是否需要同步

2. **本页面内修改顺序**
   - 确认本页面内是否有模块间的引用关系

**改后必须完成：**

1. 相关页面同步
2. CHANGE_CONTROL.md + REVIEW_LOG.md 更新

**示例场景：** 修改 home-M08（核心三页默认闭环路径）
- 先查依赖：影响 brief-M04（看完快报后怎么办）和 ops-M09（看完运营页后可以停吗）
- 改之后：同步修改 brief-M04 和 ops-M09

---

### 3.3 P2 模块维护要求

**改前必须完成：**
- 确认本模块是否被其他模块引用为依据

**改后必须完成：**
- CHANGE_CONTROL.md 更新
- REVIEW_LOG.md 更新

**不需要：**
- 不需要全站同步
- 不需要查 OPS_CHANGE_IMPACT

**示例场景：** 修改 evidence-M08（证据页训练提示）
- 本页面内修改即可
- 更新 CHANGE_CONTROL.md 和 REVIEW_LOG.md

---

### 3.4 P3 模块维护要求

**维护原则：** 观察优先，不一定需要修改

**维护要求：**
- 记录观察结论（如"此模块与某某模块近重复，可考虑未来整合"）
- 如确需修改，按 P2 模块要求处理

## 五、模块压缩前要满足什么条件

以下情况才能考虑压缩 P3 模块：

**必要条件：**
1. 该模块与另一模块高度重复（相似度 ≥80%）
2. 压缩后用户仍能找到所需信息
3. 压缩不影响任何判断逻辑
4. 有充分测试表明压缩后用户不会误用

**禁止压缩的情况：**
- P0 模块禁止压缩
- P1 模块压缩前必须完成全站评估
- 任何影响判断逻辑的模块禁止压缩
- 任何全局标准句的展示模块禁止压缩

**本轮 P3-42 结论：**
经过分析，当前所有 P3 模块都不满足压缩条件。
本轮不进行任何压缩操作，仅建立文档和标注。

## 七、模块维护的常见错误

### 错误1：改了 P0 模块但只改了一个页面
**后果：** 全站其他引用处仍显示旧内容，造成系统性不一致
**正确做法：** P0 模块改动必须全站同步

### 错误2：改了标准句但没有更新 OPS_COPY_BASELINE_V1.md
**后果：** 标准句库与实际页面内容不一致，维护者失去参照
**正确做法：** 任何标准句改动必须同步更新 OPS_COPY_BASELINE_V1.md

### 错误3：改了高依赖模块但没有更新 OPS_MODULE_DEPENDENCY_MAP
**后果：** 依赖图与实际不符，后续维护者会错误判断影响范围
**正确做法：** 任何依赖关系变化必须同步更新 OPS_MODULE_DEPENDENCY_MAP_V1.md

### 错误4：改了 P3 模块但没有更新 REVIEW_LOG
**后果：** 改动历史不完整，复盘时无法追溯
**正确做法：** 任何改动都必须更新 REVIEW_LOG（即使是 P3）

### 错误5：把"页面级改动"当成"P2 局部改动"
**后果：** 遗漏了跨页同步
**正确做法：** 修改前先用依赖图确认影响范围
