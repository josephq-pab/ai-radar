### 为什么当前必须转向系统化建设

**事实**：
- V7 作为内容基线已通过修复验证（T1/T2/T3 全部满足）
- V7 若停在此处，项目永远只是"一个月发一次的静态邮件"
- 真正限制项目价值的是系统层缺口：
  - 外部采集范围不够（9个来源，仅银行分析师）

  - 系统闭环未以完整功能系统形态收口

**判断**：V7 内容基线是必要条件，但不是充分条件——没有系统层支撑，V7 无法持续运行。

## 二、"完整功能系统 MVP"定义

**MVP 性质**：最小可用完整闭环系统——不是大而全平台，不是正式运营系统，是"能独立完成月度分析闭环"的最小系统。

**MVP 完成标准**：
- 能完成：来源配置 → 抓取 → 内容生成 → 阅读输出 → 留痕 的完整链路
- 能回答：当前启用了哪些来源 / 最近一次抓取结果如何 / 本月相比上月有哪些变化
- 能支持：每月1次人工驱动的分析产出，不依赖额外人工

## 四、模块完成标准（本期纳入）

### M1：来源注册与启停

- analyst_sources.json 是唯一配置源
- 每次变更记录到配置日志
- 变更后 fetch 验证成功
- **验收**：增删来源不需要改脚本，只改配置文件

### M2：维度配置与抓取范围扩展

- 来源数量：9个 → 15~20个
- 维度覆盖：银行分析师 + 宏观分析师 + 债市分析师
- 新增来源必须通过 dry-run 验证
- **验收**：新增来源可正常抓取并出现在 change-summary 中

### M3：统一配置入口

- 单页 HTML（可集成到 index.html）
- 展示：当前来源列表 / Gate 状态 / preflight 结果 / 上次运行时间
- 不做数据库，后端用 JSON 文件读取
- **验收**：运营者打开页面能在10秒内回答"当前系统状态是否正常"

### M4：统一运行入口

- run-analyst.sh：完整链路（fetch → build → 输出）
- run-analyst-fetch.sh：仅抓取
- 每次运行输出简要摘要（成功/失败/抓取数量）
- **验收**：一个命令完成月度闭环，不需要记忆多个脚本

### M5：统一阅读入口

- index.html 作为主入口
- 各分析维度（存款/贷款/同业/宏观）从统一入口可达
- **验收**：新用户从 index.html 出发，能在3步内找到自己要的内容

### M6：五层阅读

- 结论层：月度最重要变化（bullet point）
- 动作层：企金主办动作建议（三步式）
- 证据层：每个动作对应到具体分析师来源
- 来源层：每个分析师的活跃度和置信度
- 变化层：本月 vs 上月的关键差异
- **验收**：执行者读执行版时，能在3秒内找到"我今天要做什么"

### M7：留痕与变更控制

- 每次配置变更记录：变更内容 / 时间 / 变更人（手动记录）
- 每次运行生成 change-summary.md
- REVIEW_LOG / CHANGE_CONTROL 持续更新
- **验收**：任何人看到 change-summary，能回答"这个月和上个月有什么不同"

### M8：月度试用运行机制

- 已由 PILOT_CYCLE_PLAN_V1.md 定义
- 本期继续按该节奏运行
- **验收**：每月1次运行+发送+反馈回收的闭环已建立
