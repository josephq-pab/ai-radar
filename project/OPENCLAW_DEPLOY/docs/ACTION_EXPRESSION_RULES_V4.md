## 一、核心概念：动作闭环三角

每一个动作建议都必须回答三个问题：

1. **谁来做？** → 责任人（部门 + 角色）
2. **做完交给谁？** → 接收方（仅研究准备动作需要）
3. **什么条件下要接着做？** → 转动作条件（仅研究准备动作需要）

---

## 二、动作类型及闭环要求（V4 完整版）

| 类型 | 定义 | 闭环终点 | 是否需要接收方 | 是否需要转动作条件 |
|------|------|---------|--------------|-----------------|
| **立即执行** | 本月必须有人推动 | 完成后确认验证指标 | ❌ 不需要 | ❌ 不需要 |
| **研究准备** | 本月需要搞清楚一件事，结果给下月提供判断依据 | 接收方看完结论并决定是否触发 | ✅ 需要 | ✅ 需要 |
| **建立机制** | 本月启动但本月完不成 | 机制进入常态运行 | ⚠️ 视情况 | ⚠️ 视情况 |

## 三、研究准备动作的标准格式（V4 完整版）

```
▸ 研究准备：[研究主题]
    研究目的：[搞清楚这个问题是为了支持什么判断]
    研究责任人：企金部门（注：企金部/战略研究部均可承接，视各行实际） ← V4升级为当前推荐
    研究内容：
      1. [具体要拿到的证据/数据/结论 1]
      2. [具体要拿到的证据/数据/结论 2]
    产出形式：一页书面判断报告（3~5个要点）【打头第一条结论必须最核心】
    接收方：[部门] / [角色]
    转动作条件：
      若[触发事实]达到[判断门槛]，则下月启动[对应动作]
    预计完成时点：XXXX年XX月XX日
```

## 四、责任归口要求（V4）

### 4.1 粒度要求（V4 FB-03后降级为试行）

**V4原版**：必须到"部门 + 角色"两级 [多源确认]
**FB-03后修订**：从"多源确认"降级为**试行·三方分歧**

| 粒度级别 | 反馈来源 | 说明 |
|---------|---------|------|
| 部门 + 角色两级 | FB-01（A）、FB-02（A）| 认为能落到具体执行人 |
| 部门一级 | FB-03（C）| 认为太细，部门就够 |

**V4试行写法**：
```
责任粒度：
  建议：部门 + 角色两级（能让动作落到具体执行人）
  弹性：若本行组织架构扁平、可由部门统一部署，可简化为"部门"一级
  [试行·三份反馈存在分歧，各行可根据实际组织架构选择]
```

### 4.2 企金部门归口（V4 升级）

V3：企金部（主办）+ 战略研究部（支持）**[试行]**
V4：**企金部门（注：企金部/战略研究部均可承接，视各行实际）** [当前推荐表达]

**升级原因**：FB-01（Q1选B）+ FB-02（Q1选B）多源一致

### 4.3 降调写法

```
企金部门（注：企金部/战略研究部均可承接，视各行实际）
若本行无战略研究部，可由企金部产品经理兼任研究职能）
```

## 五、接收方的写法规则（V2/V3已定义，V4不变）

接收方 = 能决定"这个结论要不要触发下一步动作"的人，一般是部门负责人或分行长级别。

## 六、转动作条件的写法规则（V2/V3已定义，V4不变）

每个转动作条件必须包含三要素：触发事实 + 判断门槛 + 对应动作。

## 八、双门槛规则（V4 确认版）

### 8.1 结构确认

V3的双门槛结构（1000万快速扫描+5000万重点排查）经FB-02确认5000万合理后，升级为**当前推荐结构**：

| 层级 | 门槛 | 目的 | 执行要求 |
|------|------|------|---------|
| 重点排查 | 5000万以上 | 防挖转 | 必须联系，确认续做意向 |
| 快速扫描 | 1000万以上 | 摸底异常 | 不必全部联系，重点是有无异常信号 |

### 8.2 门槛弹性

**门槛数值为建议参考值**，各行可按实际存量客户数量和执行承载力调整：

```
【注】：5000万/1000万为参考值，各行可按实际客户规模和执行承载力上下浮动。
收到执行层反馈后，模板据此调整。
```

### 8.3 适用场景

| 动作类型 | 是否适用双门槛 |
|---------|-------------|
| 客户挖转风险排查 | ✅ 适用 |
| 客户到期续做提醒 | ✅ 适用 |
| 招行联动客户扫描 | ❌ 不适用（建立基准，不是排查）|
| 建立机制类动作 | ❌ 不适用 |

## 十二、报告打头结论鲜明要求（V4 新增）

### 12.1 Q3冲突的V4处理

FB-01（Q3选A：会看）vs FB-02（Q3选B：不一定，内容特别关键时才看）

**结论**：
- "一页书面判断报告"的格式本身没被否定
- 但FB-02的要求是合理的：**"内容特别关键时才看"意味着打头第一条结论必须足够关键**
- V4在产出格式要求中新增：**打头第一条结论必须是最核心的判断**

### 12.2 打头结论判断标准

**✅ 正确示范：**
```
判断句打头：华夏银行当月新增存款+558亿，增速领先但规模仍小
```

**❌ 错误示范：**
```
背景铺垫打头：两会后项目融资逐步落地，对公结算存款同步增加
```

### 12.3 格式要求

```
产出形式：一页书面判断报告（3~5个要点）
【要求】：打头第一条结论必须是最核心的判断（不是背景铺垫），占报告总篇幅不超过3行
```

## 十三、"多源确认"表达定义（V4 新增）

### 13.1 什么是多源确认

- **单源确认**：仅1份反馈来源确认某一表达（V3阶段）
- **多源确认**：≥2份独立反馈来源一致选择同一选项，无冲突（V4阶段）

### 13.2 多源确认的标注方式

在文档中使用"**[当前推荐表达]**"或"**[多源确认]**"标注：

```
研究责任人：企金部门（注：企金部/战略研究部均可承接）[当前推荐·多源确认]
```

### 13.3 多源确认不等于最终定型

- 多源确认 = "已有2+份反馈一致支持，当前最稳的表达"
- 最终定型 = "经过多轮/多机构验证，已固化不再调整"

V4的"当前推荐表达"仍是试用期的推荐，不是最终定型。

## 十四、V3试行→V4推荐的升级路径（V4 新增）

| 升级条件 | 说明 |
|---------|------|
| 反馈数量 | ≥2份独立来源 |
| 一致性 | 无冲突（同选A或同选B）|
| 覆盖层位 | 至少覆盖1个领导层或2个不同执行层背景 |
| 时间跨度 | 反馈收集时间跨度≤3个月 |

## 十五、V4 检查表（11问）

写完一条动作后自检：

1. 动作类型是否明确？
2. 动作是否具体？
3. 责任是否到部门+角色？
4. 企金部门归口是否写为"当前推荐"？
5. 是否有可验证指标？
6. 是否区分了"立即执行/研究准备"？
7. 研究准备是否有接收方？
8. 研究准备是否有转动作条件？
9. 双门槛是否加注"建议参考值，各行可调整"？
10. 报告打头第一条结论是否够核心？
11. 是否有明确的完成时点？
